## 1. 实现的 AI 能力

辅助诊断：[AIDiagnosisService](file:///d:/shixun/yunnao/backend/src/main/java/com/neusoft/cloudbrain/ai/api/AIDiagnosisService.java) 输出候选诊断、证据、缺失信息和检查建议（6 字段），不写入医生正式诊断。AI 原始结果只进 AIInvocation，候选诊断通过 `EncounterDiagnosis(source=AI_SUGGESTION)` 隔离，AI 不创建 `FINAL + DOCTOR` 记录。

## 2. 输入输出 Schema

### 输入（[DiagnosisAIRequest](file:///d:/shixun/yunnao/backend/src/main/java/com/neusoft/cloudbrain/ai/dto/DiagnosisAIRequest.java)）

| 字段 | 类型 | 说明 |
|---|---|---|
| chiefComplaint | String | 主诉 |
| presentIllness | String | 现病史 |
| pastHistory | String | 既往史 |
| physicalExamination | String | 体格检查 |
| patientAgeRange | String | 年龄段 |
| patientGender | String | 性别 |

不包含患者 ID、姓名、手机号等隐私信息（最小化原则）。

### 输出（[DiagnosisAIResult](file:///d:/shixun/yunnao/backend/src/main/java/com/neusoft/cloudbrain/ai/dto/DiagnosisAIResult.java)）

| 字段 | 类型 | 受控规则 |
|---|---|---|
| possibleDiagnoses | List\<PossibleDiagnosis\> | 候选诊断列表（按可能性排序） |
| evidence | List\<String\> | 支持诊断的证据要点（来自输入，不得编造） |
| missingInformation | List\<String\> | 建议补充的信息（无缺失时为空数组） |
| riskFactors | List\<String\> | 相关风险因素 |
| suggestedExaminations | List\<String\> | 建议的检查项目 |
| disclaimer | String | 固定免责声明 |

**PossibleDiagnosis 子结构**（仅 4 字段，不含 source/type/confirmedAt）：

| 字段 | 类型 | 说明 |
|---|---|---|
| diagnosisCode | String | ICD-10 诊断编码 |
| diagnosisName | String | 诊断名称 |
| confidence | String | 置信度枚举（HIGH / MEDIUM / LOW） |
| explanation | String | 判断依据简要说明 |

**source 隔离关键设计**：DiagnosisAIResult 和 PossibleDiagnosis 均不含 source、type、confirmedAt 等业务字段，从数据结构层面保证 AI 无法产出 `source=DOCTOR / type=FINAL` 记录。source 隔离由 [EncounterService](file:///d:/shixun/yunnao/backend/src/main/java/com/neusoft/cloudbrain/encounter/service/EncounterService.java#L269) 在写入业务表时强制（AI 只能创建 `PRELIMINARY + AI_SUGGESTION`，FINAL+DOCTOR 被拒绝）。

## 3. Prompt 文件

[diagnosis_v1.txt](file:///d:/shixun/yunnao/backend/src/main/resources/prompts/diagnosis_v1.txt)

本次更新内容（对照 [32_AI能力契约规范.md 第6节](file:///d:/shixun/yunnao/智慧云脑诊疗平台_AI协作文档_v2/contracts/32_AI能力契约规范.md)）：

- 明确任务边界（生成候选诊断列表和检查建议，供医生参考）
- 指定输出 JSON Schema（6 字段 + PossibleDiagnosis 4 子字段）
- 禁止编造患者未提及的病史、检查结果或用药史
- 诊断编码使用 ICD-10 标准，不得自行编造
- 高危症状标注 HIGH 置信度并建议紧急检查

- 信息不足时 confidence 为 LOW 并在 missingInformation 列出缺失内容
- 输入为空或信息严重不足时，possibleDiagnoses 返回空数组或仅含"病因未明"低置信度候选，不编造诊断
- 不得输出个人隐私信息和数据库 ID、医生 ID
- 输出纯 JSON，不包含 markdown 代码块标记

## 4. Provider 变化

### AIDiagnosisServiceImpl 改造

[AIDiagnosisServiceImpl](file:///d:/shixun/yunnao/backend/src/main/java/com/neusoft/cloudbrain/ai/application/AIDiagnosisServiceImpl.java) 现状（STAGE-AI-1 已完成基础重构，本任务确认对齐契约）：

1. **Schema 校验**：`parseDiagnosisResponse` 校验 3 个必填字段（possibleDiagnoses、evidence、disclaimer），解析 3 个数组字段（missingInformation、riskFactors、suggestedExaminations，缺失时返回空列表）
2. **confidence 枚举校验**：仅允许 HIGH / MEDIUM / LOW，非法值抛 AIInvalidResponseException
3. **错误码**：
   - 超时/Provider 异常 → `AI_DIAGNOSIS_FAILED`（HTTP 504）
   - 非法 JSON / Schema 校验失败 / 非法枚举 → `AI_DIAGNOSIS_FAILED`（HTTP 500）
4. **诊断隔离**：AI 结果只返回 DiagnosisAIResult（6 字段），不包含 source/type，由 encounter 业务模块强制隔离写入

### DiagnosisAIResult DTO

[DiagnosisAIResult](file:///d:/shixun/yunnao/backend/src/main/java/com/neusoft/cloudbrain/ai/dto/DiagnosisAIResult.java) 现状符合契约，无业务模块依赖（grep 确认仅 ai/** 引用），无需向后兼容构造函数。

### MockAIProvider

[MockAIProvider](file:///d:/shixun/yunnao/backend/src/main/java/com/neusoft/cloudbrain/ai/provider/MockAIProvider.java) 诊断场景现状符合契约（6 字段 Schema），已覆盖：
- 正常响应（buildDiagnosisResponse）：胸痛/头痛/发热/默认四路径，各路径均输出 6 字段
- 高风险响应（buildHighRiskDiagnosisResponse）：急性心肌梗死 HIGH 置信度 + disclaimer
- 空结果响应（buildEmptyResponse）：possibleDiagnoses 空数组 + missingInformation 列出"信息不足"

## 6. 诊断隔离策略

```text
AI 输出（DiagnosisAIResult，6 字段，无 source/type）
  → AIInvocation 记录原始结果
  → encounter 业务模块调用 EncounterService.addAIDiagnosis(encounterId, request)
  → EncounterService 强制 source=AI_SUGGESTION, type=PRELIMINARY
  → AI 尝试 DOCTOR / FINAL+AI_SUGGESTION → ENCOUNTER_DIAGNOSIS_ISOLATION_VIOLATION (409)
  → 医生最终诊断走 addDoctorDiagnosis（必须 FINAL+DOCTOR，@Auditable 审计）
```

- AI 原始结果只进 AIInvocation，不直接写业务表
- 候选诊断通过 `EncounterDiagnosis(source=AI_SUGGESTION, type=PRELIMINARY)` 隔离
- AI 不创建 `FINAL + DOCTOR` 记录（由 EncounterService 校验拦截）
- 医生诊断保存为 `source=DOCTOR`，两类记录不得互相覆盖
- AI 候选诊断不设置 confirmedAt（仅为建议，非正式确认）

## 7. 超时和降级策略

```text
超时 → 重试 1 次 → 仍失败 → AIInvocation(FAILED) → 抛 AI_DIAGNOSIS_FAILED (HTTP 504) → 业务不影响
非法 JSON → AI_DIAGNOSIS_FAILED (HTTP 500) → 不无限修复 → 降级
缺失必填字段 → AIInvalidResponseException → AI_DIAGNOSIS_FAILED (HTTP 500) → 降级
非法 confidence 枚举 → AIInvalidResponseException → AI_DIAGNOSIS_FAILED (HTTP 500) → 降级
```

- 重试仅对超时/5xx，非法 JSON 不重试
- 不无限修复模型输出
- 1 次业务调用 = 1 条 AIInvocation + N 条 AIInvocationAttempt
- AI 诊断失败不影响就诊主流程（由 encounter 业务模块捕获处理）

## 10. 对后端业务模块的调用说明

- AI 诊断不直接对外暴露 HTTP，由 encounter 业务模块编排调用
- 调用入口：`AIDiagnosisService.analyze(DiagnosisAIRequest)`
- 输入不含患者隐私 ID（最小化原则）
- AI 结果返回 DiagnosisAIResult（6 字段），由 encounter 业务模块转换为 `EncounterDiagnosisRequest(source=AI_SUGGESTION, type=PRELIMINARY)` 写入业务表
- source 隔离由 EncounterService.addAIDiagnosis 强制（AI 只能 AI_SUGGESTION，不能 DOCTOR；不能 FINAL+AI_SUGGESTION）

---

## 11. 潜在安全风险

- Prompt 中明确禁止编造和不得写入正式诊断，但真实模型可能不遵守 → 由 JsonSchemaParser 做必填字段校验 + confidence 枚举校验兜底，source 隔离由业务模块强制
- 输入已做脱敏（不含患者隐私 ID），但需业务模块确保传入前完成脱敏
- 日志和 AIInvocation 中不保存 API Key（由 AIInvocationRecorder 保证）

---

## 12. 未解决问题

- JSON Schema 校验仍为手写降级方案（必填字段 + 枚举校验），是否引入 `com.networknt:json-schema-validator` 待后端角色确认
- possibleDiagnoses 的 diagnosisCode 目前由 Mock 静态填充 ICD-10 编码，真实模型输出的编码合法性需联调验证（可考虑接入 ICD-10 编码库校验）
