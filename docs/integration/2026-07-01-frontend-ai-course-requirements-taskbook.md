# 前端 AI 任务书：课程要求补强

日期：2026-07-01  
分支：`codex/integration`  
依据文档：`docs/integration/2026-07-01-course-requirements-checklist.md`（开头标记：修改后）

## 一、任务目标

前端 AI 负责补齐课程对照文档中属于前端页面、状态管理、交互展示和用户路径的缺口。目标不是重做业务，而是在当前 Vue 3 + TypeScript + Element Plus + Axios + Pinia 项目上做最小侵入补强，让课程演示更贴合截图要求。

## 二、工作边界

允许做：

- 修改患者端、医生端相关 Vue 页面和前端 API client。
- 新增或拆分 Pinia store。
- 增加页面 loading、错误提示、空状态和跳转参数处理。
- 增加必要的 TypeScript 类型定义。
- 配合现有后端接口或后端 AI 新增的兼容接口完成联调。

不允许做：

- 不修改 OpenAPI 主契约基线，除非后端 AI 已提供兼容接口并完成说明。
- 不擅自改变现有路由权限、角色模型、数据库边界。
- 不移除已有真实联调链路。
- 不把 mock 重新引入已经接入真实接口的流程。

## 三、任务清单

### F1：分诊结果页补推荐医生列表与医生级跳转

对应课程要求：

- 任务三要求前端展示 AI 推荐的科室与医生列表。
- 支持用户一键跳转至该医生的挂号页面。

当前问题：

- 当前分诊结果页只直接展示推荐科室。
- 按钮是“去该科室挂号”，只携带 `departmentId/departmentName`。
- 医生列表是在挂号页按科室再展示，不是在分诊结果页直接展示。

前端实现要求：

- 在患者分诊结果页根据 AI 推荐科室加载可预约医生/排班。
- 展示 2-3 个推荐医生卡片，至少包含医生姓名、科室、职称/简介、可预约日期或号源摘要。
- 每张医生卡片提供“预约该医生”按钮。
- 跳转挂号页时携带 `doctorId`、`departmentId`，如已有可用 `scheduleId` 也一并携带。
- 挂号页读取跳转参数后自动预选对应科室、医生和可用排班。
- 推荐医生接口失败时，不阻塞分诊结果展示，应显示“医生列表暂不可用，可进入科室挂号”的降级入口。

验收标准：

- 分诊成功后能在结果页直接看到推荐科室和医生列表。
- 点击“预约该医生”后，挂号页能定位到该医生或至少定位到该医生所在科室并突出可选医生。
- loading、空状态、接口失败状态都有清晰提示。
- 不影响原有“去该科室挂号”路径。

### F2：医生处方页补历史处方与 AI 审核结果面板

对应课程要求：

- 任务四要求展示历史处方列表及对应 AI 审核结果，如风险等级和检测项。

当前问题：

- 医生端当前主要展示当前就诊处方及其 AI 审核结果。
- 后端已有按医生/患者查询处方能力，但医生处方页没有单独的“历史处方”展示区。
- 当前“AI 审核”按钮更像读取已有审核结果，而不是显式触发课程示例中的审核接口。

前端实现要求：

- 在医生处方页增加“历史处方”折叠区或表格区。
- 按当前患者或当前就诊上下文加载历史处方。
- 展示处方状态、创建时间、药品摘要、AI 审核状态、风险等级、关键警告。
- 如果后端 AI 提供 `/api/prescription/check` 兼容接口，则“AI 审核”按钮优先调用该接口。
- 如果后端仍采用创建处方时自动审核，则按钮文案应更准确，例如“查看 AI 审核结果”或“刷新审核结果”。
- 历史处方为空时展示空状态，不应误导为接口失败。

验收标准：

- 医生能在处方页看到当前患者既往处方及 AI 风险结果。
- 高风险、中风险、低风险有清晰视觉区分。
- 当前处方创建、审核、确认流程不被破坏。

### F3：医生病历页补“问诊对话记录”多行输入区

对应课程要求：

- 任务五要求医生端可以录入问诊对话记录，作为 AI 病历生成输入。

当前问题：

- 当前医生端主要通过结构化字段生成病历。
- 缺少模拟问诊对话文本录入区。

前端实现要求：

- 在医生病历页或接诊概览页增加“问诊对话记录”多行文本区。
- 文本区支持粘贴或手动录入医患对话。
- 调用 AI 病历生成时，将该文本作为请求参数传给后端。
- 生成成功后仍回填结构化病历字段，医生可继续编辑后保存/确认。
- 文本区为空时保持原有结构化字段生成逻辑，不阻塞现有流程。

验收标准：

- 医生能输入一段模拟医患对话并生成病历草稿。
- AI 生成失败时保留医生输入内容，并提示可手动填写。
- 不影响原有主诉、现病史、既往史、体格检查等字段。

### F4：Pinia 状态模块化补强

对应拓展要求：

- 使用 Pinia 模块化管理患者信息、医生信息、挂号记录、处方记录等状态，实现复杂交互流程状态流转。

当前问题：

- 当前已有 `auth` 和 `encounter` store。
- 患者信息、医生信息、挂号记录、处方记录仍主要散落在页面局部状态中。

前端实现要求：

- 保留现有 `auth`、`encounter` store。
- 可新增以下 store，按实际代码复用程度决定最小实现：
  - `patient`：当前患者资料、刷新患者信息。
  - `doctor`：医生列表、按科室医生列表、医生详情缓存。
  - `appointment`：我的挂号、当前预约上下文、从分诊跳转来的预选参数。
  - `prescription`：当前就诊处方、历史处方、AI 审核结果摘要。
- Store 不应变成大而全的全局缓存；只抽取跨页面复用或复杂流转状态。
- 页面迁移时优先处理与 F1/F2/F3 直接相关的数据流。

验收标准：

- 分诊到挂号的预选医生/科室状态通过 Pinia 或明确路由参数稳定流转。
- 医生处方页历史处方状态可刷新、可清空、不污染其他患者。
- 刷新页面后登录状态仍按现有机制恢复。
- `npm run build` 通过。

## 四、交付物

- 修改后的前端页面和 store。
- 前端 API client 与类型定义更新。
- 简短交付说明，列出：
  - 改了哪些页面。
  - 调用了哪些接口。
  - 哪些功能依赖后端 AI 的兼容接口。
  - 本地验证命令和结果。

## 五、联调交接点

交给联调 AI 验证的场景：

- 患者分诊后直接看到推荐医生，并可跳转挂号。
- 医生端可查看当前患者历史处方和 AI 审核结果。
- 医生端输入问诊对话记录后可生成并确认病历。
- Pinia 状态流转不导致跨用户、跨患者数据串扰。
