# 联调 AI 任务书：真实上手网页问题验收与收口

日期：2026-07-02
分支：`codex/integration`
依据文档：`docs/integration/2026-07-02-hands-on-web-feedback-issues.md`

本文档只分配给联调 AI。联调 AI 不替代前端 AI / 后端 AI 做大范围业务实现，主要负责复现、证据采集、契约同步、端到端验收、Docker/partner 环境一致性、最终取舍和提交前判断。若发现必须改前端或后端代码，可做小范围联调修复；涉及较大业务逻辑时回派给对应 AI。

## 一、总体目标

把实际上手网页暴露出的 10 个问题从“用户反馈”推进到“可复现、已分派、已修复或有明确取舍”的状态，最终保证项目能稳定演示完整诊疗闭环。

## 二、联调总原则

- 所有问题只分配给三个角色：前端 AI、后端 AI、联调 AI。
- 优先处理会阻断演示闭环的问题：处方不同步、新增医生失败、多轮 AI 无上下文、日志错误。
- 对于“是否保留”的页面，联调 AI 负责按课程要求和演示价值给出取舍。
- 每个问题必须保留最小证据：接口 URL、请求 payload、响应 body、控制台错误、数据库查询结果或自动化测试输出。

## 三、任务清单

### I-HW-02：处方医生端到患者端闭环验收

关联问题：HW-02

任务：
1. 患者挂号。
2. 医生接诊。
3. 医生开具并确认处方。
4. 患者登录。
5. 患者端查看处方列表。
6. 打开处方详情。

验收：
- 医生确认的处方在患者端可见。
- 患者端处方列表和详情均可打开。
- 若失败，明确是后端关联/状态问题还是前端字段/渲染问题。

### I-HW-03：管理员首次改密删除验收

关联问题：HW-03

任务：
- 新建或重置一个管理员账号。
- 首次登录该管理员账号。
- 验证是否直接进入管理端首页。
- 验证主动修改密码仍可用。

验收：
- 管理员首次登录不再强制跳改密。
- 管理员主动修改密码能力不受影响。

### I-HW-04：用户管理字段验收

关联问题：HW-04

任务：
- 准备管理员、医生、患者三类账号。
- 分别登录一次。
- 管理员进入用户管理页。
- 验证电话、最后登录、角色、状态显示。

验收：
- 有电话的账号显示真实电话。
- 登录过的账号显示最后登录时间。
- 三类角色显示正确。

### I-HW-05：新增医生 400 复现和修复验收

关联问题：HW-05

任务：
- 使用 6 位密码复现一次新增医生失败，记录请求 payload 和响应 body。
- 使用合法数据新增医生。
- 验证新增医生出现在医生管理列表，并能登录医生端。

验收：
- 合法新增医生成功。
- 非法新增医生失败时页面显示具体原因。
- OpenAPI 或交付说明写清字段和枚举。

### I-HW-06：患者管理页面验收

关联问题：HW-06

任务：
- 验证菜单名称为“患者管理”。
- 首次进入页面时默认展示所有患者。
- 验证姓名/账号/电话/状态筛选。
- 确认页面没有无后端支撑的假操作按钮。

验收：
- 默认列表可见。
- 筛选准确。
- 演示患者 5 人可稳定显示。

### I-HW-07：AI 分诊多轮上下文验收

关联问题：HW-07

任务：
- 使用同一个患者执行多轮分诊：
  1. 第一轮输入：“发烧三天”。
  2. 第二轮继续输入：“伴有咳嗽和胸闷”。
- 记录第二轮请求是否携带 `conversationId/history/round`。
- 记录后端日志或 AI 调用审计，确认 provider 输入包含历史上下文。
- 记录第二轮返回是否综合两轮病情。

验收：
- 第二轮请求不是新会话。
- 第二轮 AI 建议与第一轮主诉有关。
- AI 调用审计可追踪本次调用。

### I-HW-08：统计驾驶舱保留/隐藏决策与验收

关联问题：HW-08

课程对照判断：
- 项目要求包含“大数据分析技术”“诊疗质量反馈”“完整诊疗闭环”，统计页面有演示价值。
- 但核心任务仍是传统业务 + AI 分诊 + AI 处方审核 + AI 病历生成。

任务：
- 优先推动后端 AI 修复设备统计 SQL 500，前端 AI 做分块降级。
- 若修复后可稳定展示基础指标，则保留“统计驾驶舱”。
- 若演示前仍频繁 500 或数据无法解释，则要求前端 AI 临时隐藏菜单，不删除代码。

验收：
- 保留方案：统计驾驶舱稳定展示基础指标，不 500。
- 隐藏方案：管理员菜单不暴露统计驾驶舱，其他页面不受影响。

### I-HW-09：登录日志验收

关联问题：HW-09

任务：
- 用管理员、医生、患者分别登录。
- 进入登录日志页。
- 验证用户名、角色、登录时间、登录结果。
- 确认 IP 地址列已删除。

验收：
- 不再大量显示 `SYSTEM / SYSTEM`。
- 角色不是统一 `USER`，而是能区分管理员、医生、患者。
- IP 地址不在表格展示。

### I-HW-10：操作日志验收

关联问题：HW-10

任务：
- 触发至少三类操作：
  - 新增医生。
  - 重置密码。
  - 修改排班或禁用账号。
- 进入操作日志页。
- 验证操作人、目标类型、目标 ID/目标名称、操作内容。

验收：
- 日志能读出“谁对哪个对象做了什么”。
- 目标类型不展示难懂内部枚举。
- 目标 ID 不错位。

### I-HW-11：AI 调用记录供应商和模型验收

关联问题：HW-11

任务：
- 使用真实 DeepSeek key 触发至少一次 AI 调用。
- 进入 AI 调用记录页。
- 验证供应商显示 `DeepSeek`。
- 验证模型显示 `v4 flash`。
- 再用 mock provider 或无 key 场景确认 mock 记录不会伪装为 DeepSeek。

验收：
- 真实 AI 调用记录显示 `DeepSeek` / `v4 flash`。
- Mock 和真实调用可区分。

## 四、OpenAPI 与契约同步任务

联调 AI 负责在前后端修复后核对并同步：
- 用户管理列表字段。
- 新增医生请求/响应和错误响应。
- 患者处方列表/详情响应字段。
- 分诊多轮字段：`conversationId`、`history`、`round`。
- 统计驾驶舱响应结构。
- 登录日志、操作日志、AI 调用记录字段。

验收：
- OpenAPI 与后端 Controller / DTO 一致。
- 前端 API client 类型与 OpenAPI/后端响应一致。

## 五、Docker 与 partner 环境验收

任务：
- 确认 partner 已有 `.env` 或能从 `.env.example` 正确复制。
- 确认 partner 能拉取 MySQL 镜像，或给出可用替代方案。
- 确认 partner 清理旧 volume 后可重新初始化演示数据。
- 确认 `docker compose up -d mysql backend frontend` 后三端可访问。

验收：
- partner 本地可从 GitHub 分支启动项目。
- 演示账号可登录。
- 真实闭环可跑通。

## 六、端到端最终验收

必须覆盖以下完整场景：

1. 患者注册/登录。
2. 患者 AI 多轮分诊。
3. 患者根据推荐科室/医生挂号。
4. 医生登录并接诊。
5. 医生生成或填写病历。
6. 医生开检查、查看检查状态。
7. 医生开处方并触发 AI 审核。
8. 患者端查看处方和病历。
9. 管理员查看用户管理、患者管理、医生管理、登录日志、操作日志、AI 调用记录。

验收：
- 主链路不白屏、不 500、不展示明显假数据或错误角色。
- AI 能力和传统业务能形成闭环。

## 七、交付说明要求

联调 AI 最终输出交付报告，必须包含：
- 10 个 HW 问题的最终状态：已修复 / 已隐藏 / 待后续 / 无法复现。
- 每个问题的接口或测试证据。
- 前端 AI、后端 AI 的交付核对结果。
- OpenAPI 同步结果。
- Docker 和 partner 环境验证结果。
- 最终是否可以提交 GitHub 的判断。
