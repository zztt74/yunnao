# 后端 AI 任务书：真实上手网页问题修复

日期：2026-07-02
分支：`codex/integration`
依据文档：`docs/integration/2026-07-02-hands-on-web-feedback-issues.md`

本文档只分配给后端 AI。后端 AI 负责接口行为、服务逻辑、字段聚合、状态流转、日志审计、错误响应、AI provider 审计记录和后端测试；不得擅自重建数据库基线或修改治理文档。需要数据库字段变更时，使用增量迁移并说明兼容影响。

## 一、总体目标

修复实际上手网页中暴露出的后端数据、接口、状态、审计和 AI 上下文问题，保证前端三端在真实后端下能稳定联动，避免出现“前端有页面但后端数据不通”的演示断点。

## 二、任务清单

### B-HW-02：处方创建后患者端可见

关联问题：HW-02

已定因：
- 患者处方列表响应缺少医生、科室、患者等展示字段，导致前端需要猜或填空。
- 患者端可见性还受处方状态影响，DRAFT 不应被患者误认为已同步失败。

任务：
- 核对处方创建、确认、患者列表、患者详情接口的患者关联、状态过滤和权限。
- 明确患者端可见状态，例如 CONFIRMED、DISPENSED、VOIDED 的展示规则。
- 患者列表和详情接口返回医生姓名、科室、患者姓名、药品明细、开具时间、审核结果等必要展示字段。
- 无权限访问其他患者处方时返回明确 403。

验收：
- 医生给患者开具并确认处方后，该患者登录患者端能查询到该处方。
- 患者端处方详情字段完整，前端不需要依赖 encounter 上下文补字段。

交接：
- 字段新增后同步 OpenAPI、前端类型和联调验收脚本。

### B-HW-03：关闭管理员首次登录强制改密

关联问题：HW-03

已定因：
- `mustChangePassword` 机制仍贯穿登录响应、JWT 拦截和重置密码流程。

任务：
- 移除或关闭管理员首次登录强制改密校验。
- 登录响应不再要求管理员首次登录改密。
- 权限拦截器不再因管理员 `mustChangePassword` 拦截管理端接口。
- 重置密码后不再强制管理员下次改密。
- 保留用户主动修改密码能力。

验收：
- 新管理员账号首次登录后可直接访问管理端首页接口。
- 重置管理员密码后再次登录不被强制改密。

### B-HW-04：用户管理列表补齐电话和最后登录

关联问题：HW-04

已定因：
- `AdminUserResponse` 不包含 `lastLoginAt`。
- `UserAccount` 没有最近登录时间字段或聚合来源。
- 医生创建流程未把表单中的电话/邮箱同步写入账号表。

任务：
- 登录成功后记录最近登录时间，或从登录日志聚合最近成功登录时间。
- 用户管理列表聚合管理员、医生、患者的手机号。
- 返回字段至少包含 username、displayName、role、phone、lastLoginAt、status。
- 新增医生时同步账号表需要展示的电话/邮箱/姓名。
- 后端不要返回 `--`、`从未登录` 等前端展示文案。

验收：
- 已配置手机号的用户接口返回真实手机号。
- 成功登录后的用户接口返回最近登录时间。
- 管理员、医生、患者三类用户均可正确聚合。

### B-HW-05：新增医生接口 400 与错误响应

关联问题：HW-05

已定因：
- 后端要求密码 8-64 位，前端曾按 6 位放行。

任务：
- 保持或明确医生初始密码规则，并同步 OpenAPI。
- 对科室、职称、性别、手机号、初始密码、账号唯一性给出明确校验错误。
- 错误响应必须包含可读 `message`，不能只返回 400。
- 合法输入必须能成功新增医生，并建立账号、医生档案和科室关联。

验收：
- 合法新增医生请求返回成功。
- 非法请求返回 400 且包含明确错误原因。
- 新增医生后能在医生管理列表出现，并可登录医生端。

### B-HW-06：患者管理列表能力确认

关联问题：HW-06

任务：
- 管理端患者列表接口默认返回全部患者。
- 支持分页。
- 支持关键字筛选，至少覆盖姓名、账号、电话。
- 支持状态筛选。
- 返回患者管理页需要的基础信息。

验收：
- 管理员无筛选调用接口能获取全部演示患者。
- 关键字和状态筛选可用。
- 分页 total 与 records 一致。

### B-HW-07：AI 分诊多轮上下文真实生效

关联问题：HW-07

已定因：
- `AITriageService` 多轮默认实现直接调用单轮方法并忽略 history。
- `AITriageServiceImpl` 未覆盖多轮重载，历史对话在 provider prompt 前被丢弃。

任务：
- 覆盖实现多轮 `analyze(request, history, round)`。
- DeepSeek provider 请求必须包含历史对话上下文，不能只传最新一句。
- Mock provider 也应模拟多轮上下文，便于测试。
- 第二轮返回结果应综合前后两轮信息。
- 若增加服务端会话存储，必须用增量迁移并说明兼容性。

验收：
- 单元测试覆盖：第一轮“发烧三天”，第二轮“伴咳嗽胸闷”，第二轮 provider 输入包含两轮内容。
- 第二轮建议与第一轮主诉有关联。

### B-HW-08：统计驾驶舱接口 500 修复

关联问题：HW-08

已定因：
- `StatisticsRepository.getDeviceUsageStatistics()` 查询 `device_status_history.created_at`，但表结构实际字段是 `changed_at`。
- 该接口 500 会被前端统计页放大成整页失败。

任务：
- 修复设备统计 SQL 字段错误。
- 统计接口在空数据、日期为空、科室为空、聚合结果为空时不得 500。
- 至少稳定返回基础指标：总患者数、今日挂号数、待诊人数、接诊中人数、已完成就诊数、AI 调用数。
- 指标暂缺真实来源时返回 0 或空数组，不得抛异常。

验收：
- 管理端统计驾驶舱接口在空库、演示库、有筛选条件时均不 500。
- 设备统计接口单独调用成功。

### B-HW-09：登录日志真实记录用户名和角色

关联问题：HW-09

已定因：
- 登录方法执行审计时安全上下文尚未建立，`AuditAspect` 记录为 SYSTEM。
- 已认证请求的 `operatorType` 统一为 USER，不区分 ADMIN/DOCTOR/PATIENT。

任务：
- 登录成功日志记录真实 username、displayName、role。
- 避免普通用户登录被记录成 `SYSTEM`。
- 角色不要统一记为 `USER`，应能区分 ADMIN、DOCTOR、PATIENT。
- IP 字段可继续保留给审计，但不要求前端展示。
- 失败登录尽量记录输入用户名和失败原因。

验收：
- 管理员、医生、患者分别登录后，日志接口返回正确用户名和角色。
- 登录日志列表不再大量出现无意义 `SYSTEM / SYSTEM`。

### B-HW-10：操作日志语义补齐

关联问题：HW-10

已定因：
- `AuditAspect` 用 `targetIdParam` 或第一个 Long 参数启发式取目标 ID，DTO 请求场景容易为空或不准。
- 操作人类型只有 USER/SYSTEM，缺少真实角色语义。

任务：
- 操作日志记录真实操作人、角色、操作类型、目标类型、目标 ID。
- 尽量补充目标名称或摘要，例如医生姓名、患者姓名、排班日期、处方编号。
- 管理员敏感操作必须留痕，例如新增医生、修改用户、重置密码、禁用账号。
- 区分系统自动操作和人工操作，不混用 `SYSTEM`。

验收：
- 执行新增医生、重置密码、修改排班等操作后，操作日志能定位操作人和目标对象。
- 接口返回字段足够前端展示“谁在什么时候对哪个对象做了什么”。

### B-HW-11：AI 调用审计供应商和模型

关联问题：HW-11

已定因：
- `AIInvocationResponse` 列表 DTO 不包含 provider/model；字段只存在于 attempt DTO。

任务：
- AI 调用记录列表和详情接口返回 provider display name 和 model display name。
- 真实 DeepSeek 调用写入供应商 `DeepSeek`，模型 `v4 flash`。
- Mock provider 调用不能写成 DeepSeek，应显示 Mock 或本地模拟。
- 返回能力类型、是否成功、耗时、错误原因、最近一次 attempt provider/model。

验收：
- 使用真实 DeepSeek provider 调用一次后，审计记录显示 `DeepSeek` / `v4 flash`。
- 使用 mock provider 调用时，审计记录能区分 mock。
