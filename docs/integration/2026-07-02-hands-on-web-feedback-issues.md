## 二、逐项记录

### HW-02：医生开具处方后患者端不同步或处方页空白

现象：
- 医生开具处方后，患者端处方列表没有及时显示。
- 有时患者端处方页面打不开，或打开后没有界面内容。

初步判断：
- 这是完整闭环问题，不能只按前端展示处理。
- 可能来源包括：
  - 医生端处方保存接口成功，但患者端查询接口未按患者 ID / encounter ID 关联到同一条处方。
  - 患者端 API client 字段映射与后端响应不一致，导致页面渲染异常。
  - 处方状态未从 `DRAFT` / `SUBMITTED` / `CONFIRMED` 正确流转到患者可见状态。
  - 前端页面对空数据或异常响应没有兜底，导致空白页。

责任划分：
- 后端 AI：
  - 核对处方创建、确认、患者查询三段接口的患者关联、状态过滤和权限。
  - 明确患者端可见哪些处方状态。
- 前端 AI：
  - 修复患者端处方页空白兜底。
  - 核对患者端处方 API 字段映射。
  - 若处方保存后需要刷新患者端列表，补齐页面刷新逻辑。
- 联调 AI：
  - 增加闭环验收：医生开处方 -> 患者登录 -> 处方列表可见 -> 处方详情可打开。

验收口径：
- 同一患者就诊中，医生确认处方后，患者端无需改数据库即可看到对应处方。
- 患者端处方页面在无数据、接口 401/403/500、字段缺失时不出现白屏。

### HW-03：删除管理员首次登录强制改密功能

现象：
- 用户明确要求：管理员登录首次改密功能直接删除。

初步判断：
- 这是产品交互调整，不是缺陷修复。
- 如果后端存在 `mustChangePassword`、`passwordChangedAt`、首次登录拦截等逻辑，需要前后端同时处理，否则只删前端入口可能仍被后端拦截。

责任划分：
- 前端 AI：
  - 删除或绕过管理员首次登录强制改密页面和跳转逻辑。
  - 保留管理员主动修改密码能力即可。
- 后端 AI：
  - 移除或关闭管理员首次登录强制改密校验。
  - 确认登录响应中不再要求前端跳改密。
- 联调 AI：
  - 验收全新管理员账号首次登录后能直接进入管理端首页。

验收口径：
- 新增或重置后的管理员账号首次登录，不再被强制跳转到改密页。

### HW-04：管理员用户管理电话和最后登录显示不对

现象：
- 管理员用户管理表格中，电话列大量显示 `--`。
- 最后登录列大量显示“从未登录”，与实际登录情况可能不一致。

初步判断：
- 电话问题可能是账号表与患者/医生 profile 表字段没有聚合，或前端取错字段。
- 最后登录问题可能是登录成功后未回写用户表，或日志表有记录但用户列表接口未聚合。

责任划分：
- 后端 AI：
  - 明确用户管理列表接口的电话来源：患者 profile、医生 profile、管理员账号资料。
  - 登录成功后更新 `lastLoginAt`，或列表接口从登录日志聚合最近成功登录时间。
- 前端 AI：
  - 修正字段映射，电话和最后登录为空时只在真实为空时显示 `--`/“从未登录”。
- 联调 AI：
  - 用管理员、医生、患者各登录一次后，刷新用户管理列表验证最后登录。

验收口径：
- 已有手机号的医生/患者/管理员在用户管理列表显示真实手机号。
- 成功登录过的账号不再显示“从未登录”。

### HW-05：管理员新增医生返回 400

现象：
- 管理员端新增医生弹窗提交后，页面提示 `Request failed with status code 400`。

初步判断：
- 需要查看接口响应体才能定因。
- 截图中可能触发 400 的字段包括：
  - 初始密码强度不满足后端校验。
  - 职称枚举前端提交中文，后端期望枚举值。
  - 科室前端提交科室名称，后端期望 departmentId。
  - 手机号格式或唯一性校验失败。
  - 登录账号已存在。

责任划分：
- 前端 AI：
  - 提交失败时展示后端返回的具体错误信息，而不是通用 400。
  - 核对新增医生表单提交 payload，确保枚举、科室、性别、手机号字段符合后端契约。
- 后端 AI：
  - 确认新增医生接口校验规则和错误响应是否清晰。
  - 如果当前接口只支持内部枚举值，文档和 OpenAPI 要写清楚。
- 联调 AI：
  - 用截图中的输入重跑一次，记录请求 payload 和响应 body。

验收口径：
- 管理员使用合法信息可成功新增医生。
- 非法信息失败时，页面能显示具体原因，例如“初始密码至少 8 位并包含字母数字”或“登录账号已存在”。

### HW-06：管理员“患者查询”改为“患者管理”

现象：
- 用户要求管理员端“患者查询”改成“患者管理”。
- 页面默认直接展示所有患者信息，并提供筛选查询功能。

初步判断：
- 这是页面定位调整。
- 当前“查询”容易让人理解成只读检索页；课程/项目更需要管理端能查看患者完整资料、状态、账号情况。

责任划分：
- 前端 AI：
  - 菜单文案从“患者查询”改为“患者管理”。
  - 页面默认加载患者列表，不要求先输入搜索条件。
  - 保留姓名、账号、电话、状态、性别、年龄等筛选。
  - 根据后端能力展示编辑、禁用、重置密码等操作；没有后端接口的操作不要假按钮。
- 后端 AI：
  - 如果现有患者列表接口不支持分页、状态、姓名/电话筛选，需要补齐。
- 联调 AI：
  - 验证管理员首次进入患者管理页即可看到 5 个演示患者。

验收口径：
- 菜单名称为“患者管理”。
- 默认展示全部患者，筛选条件可正常缩小结果。

### HW-07：AI 分诊第二轮未继承第一轮记忆

现象：
- 患者端 AI 分诊第二轮对话没有继承第一轮内容。
- 第二轮给出的建议与第一轮病情缺少相关性。

初步判断：
- 这是 P1 闭环问题，直接影响课程“AI 融合”和用户体验。
- 之前已经做过多轮字段，但从用户实测看，真实请求、后端传给 AI 的上下文、或 provider prompt 组装仍可能没有真正串起来。

责任划分：
- 前端 AI：
  - 确认第二轮请求实际携带 `conversationId`、`history`、`round`、当前补充内容。
  - 页面继续对话时不能重新创建一次独立问诊。
- 后端 AI：
  - 确认 `/api/triage/consult` 或 `/api/triage/analyze` 真正把 history 拼入 AI provider 请求。
  - 确认 mock provider 和 DeepSeek provider 都按上下文生成结果。
  - 必要时增加服务端会话存储，避免完全依赖前端传 history。
- 联调 AI：
  - 设计 E2E：第一轮“发烧三天”，第二轮“伴有咳嗽和胸闷”，断言第二轮建议包含前后两轮信息。

验收口径：
- 第二轮回复能明确引用或综合第一轮主诉。
- 不允许第二轮只按最新一句话重新分诊。

### HW-08：统计驾驶舱是否必要及 500 问题

现象：
- 管理员端“统计驾驶舱”页面加载失败，返回 500。

对照课程要求的初步判断：
- 课程项目概述提到“大数据分析技术”“诊疗质量反馈”“完整诊疗闭环”，因此统计类页面有保留价值。
- 但课程核心任务主要集中在前后端 CRUD、AI 分诊、AI 处方审核、AI 病历生成、综合场景演示。统计驾驶舱不是最核心闭环入口。
- 建议结论：保留，但降级为稳定的基础统计页面；如果短时间无法修复后端 500，应临时从菜单隐藏，避免演示时暴露错误。

责任划分：
- 后端 AI：
  - 修复统计接口 500。
  - 至少返回总患者数、今日挂号数、待诊/接诊中/已完成数量、AI 调用数等基础指标。
- 前端 AI：
  - 页面接口失败时展示友好空状态，不直接暴露 500。
  - 若后端数据不足，展示“暂无统计数据”，不要白屏。
- 联调 AI：
  - 判断演示前是否保留菜单：能稳定返回则保留，不能稳定返回则隐藏。

验收口径：
- 统计驾驶舱不再 500。
- 若保留，页面至少能展示稳定的基础运营指标。

### HW-09：登录日志用户名和角色显示错误，去掉 IP 地址

现象：
- 登录日志中出现大量 `SYSTEM / SYSTEM`。
- 真实用户如 `zhang`、中文名等显示为 `USER` 角色，角色语义不准确。
- IP 地址显示 `0:0:0:0:0:0:0:1` 或类似本地 IPv6，不适合演示，用户要求直接去掉。

初步判断：
- 用户名和角色可能取自 token subject、认证 principal 或日志默认值，而不是系统账号表中的 username/displayName/role。
- 角色展示应映射为管理员、医生、患者，而不是直接显示 `USER`。
- IP 地址对课程演示价值低，且本地环境显示不好，按用户要求从表格移除。

责任划分：
- 后端 AI：
  - 登录日志记录真实账号 username、displayName、role。
  - 避免成功登录被记成 `SYSTEM`。
  - 若保留 IP 字段给审计，接口可以继续返回，但前端不展示。
- 前端 AI：
  - 表格移除 IP 地址列。
  - 角色字段映射为中文角色。
  - 用户名优先显示真实账号或姓名，不显示 `SYSTEM` 占位。
- 联调 AI：
  - 分别用 admin、医生、患者登录，验证登录日志显示正确角色。

验收口径：
- 登录日志不再把普通登录显示为 `SYSTEM / SYSTEM`。
- 表格不展示 IP 地址列。

### HW-10：操作日志字段语义有问题

现象：
- 用户反馈操作日志中的操作人、目标类型、目标 ID 有问题。

初步判断：
- 需要进一步截图或接口数据确认，但属于审计日志契约/展示问题。
- 可能原因：
  - 后端记录的 actor 使用系统默认值或 token subject 不准确。
  - targetType 使用内部枚举，前端未翻译。
  - targetId 没有关联业务对象可读名称，导致演示不可理解。

责任划分：
- 后端 AI：
  - 操作日志记录真实操作人、操作类型、目标类型、目标 ID、目标名称或摘要。
  - 明确管理员操作、医生操作、系统操作的边界。
- 前端 AI：
  - 将 targetType 翻译为中文业务对象，例如“医生”“患者”“排班”“处方”“检查检验”。
  - 如果只有 targetId，展示为次要信息；优先展示目标名称或摘要。
- 联调 AI：
  - 通过新增医生、修改排班、重置密码等操作验证日志链路。

验收口径：
- 操作日志能让演示者看懂“谁在什么时候对哪个对象做了什么”。

### HW-11：AI 调用记录供应商和模型显示 DeepSeek / v4 flash

现象：
- 用户要求 AI 调用记录中供应商显示 `DeepSeek`，模型显示 `v4 flash`。

初步判断：
- 当前 AI provider 可能记录为内部 provider key、mock、http，或模型字段为空。
- 如果实际接入的是 DeepSeek，应统一审计显示文案。
- 注意：真实 DeepSeek 官方模型名通常不是“v4 flash”这种命名；但本项目演示口径按用户要求显示为 `v4 flash`。

责任划分：
- 后端 AI：
  - AI 调用审计记录 provider display name 和 model display name。
  - DeepSeek provider 成功调用时写入 `DeepSeek` / `v4 flash`。
  - mock provider 仍应明确显示为 Mock，避免误导真实调用。
- 前端 AI：
  - AI 调用记录列表和详情展示供应商、模型字段。
  - 不要把内部枚举直接暴露给用户。
- 联调 AI：
  - 使用真实 DeepSeek key 调用一次分诊或处方审核，验证审计记录显示。

验收口径：
- 真实 AI 调用记录显示供应商 `DeepSeek`，模型 `v4 flash`。

## 五、定因结果（2026-07-02 联调 AI 复查）

本节基于当前 `codex/integration` 代码和本文档中的 10 条实际上手问题进行定因。只记录原因，不直接替代后续修复任务。

| 编号 | 定因状态 | 根因结论 | 主责 |
|---|---|---|---|
| HW-02 | 已定因 | 患者处方列表接口返回的 `PrescriptionResponse` 只含处方业务字段，不含医生姓名、科室、患者姓名等展示字段；前端 `mapPrescription()` 在无 encounter 上下文时把这些字段映射为空字符串；患者处方页又直接执行 `r.doctorName.charAt(0)`，所以医生确认处方后，患者端可能因空字段渲染异常而白屏/打不开。同步本身还受处方状态影响：医生端只保存 DRAFT 时患者端不可见，确认 CONFIRMED 后才应可见。 | 后端 AI 补列表展示字段/状态口径；前端 AI 做空值兜底和字段映射；联调 AI 验证闭环 |
| HW-03 | 已定因 | 强制改密不是只在前端残留。后端仍保留 `mustChangePassword` 机制：重置密码会设置 `mustChangePassword=true`，登录响应也返回该字段，JWT 过滤器还有 must-change 拦截逻辑；初始化管理员也可能被标记为必须改密。因此只删页面不够，必须前后端同时关闭管理员首次强制改密链路，同时保留用户主动改密。 | 后端 AI 主责；前端 AI 删除/绕过入口；联调 AI 验证新管理员首次登录 |
