# 后端 AI 任务书：课程要求补强

日期：2026-07-01  
分支：`codex/integration`  
依据文档：`docs/integration/2026-07-01-course-requirements-checklist.md`（开头标记：修改后）

## 一、任务目标

后端 AI 负责补齐课程对照文档中属于后端接口、AI 服务、兼容路径、Prompt 配置和 API 文档展示的缺口。目标是在不破坏现有真实联调闭环的前提下，补强课程展示口径。

## 二、工作边界

允许做：

- 新增兼容接口或轻量转发接口。
- 新增可选请求字段，保持向后兼容。
- 补 Knife4j `/doc.html` 文档入口。
- 扩展 AI Prompt 模板加载策略。
- 增加必要的单元测试、集成测试或接口测试。

不允许做：

- 不重建数据库基线。
- 不破坏现有 RESTful 接口。
- 不删除已通过 E2E 的真实业务链路。
- 不把 AI 结果直接作为最终医疗结论，仍需保留医生确认边界。

## 三、任务清单

### B1：补 Knife4j `/doc.html`

对应课程要求：

- 任务一要求集成 Knife4j/Swagger3，并可通过 `/doc.html` 访问 API 文档。

当前问题：

- 当前项目使用 `springdoc-openapi-starter-webmvc-ui`。
- 主要访问路径为 `/swagger-ui.html`，与课程示例 `/doc.html` 不一致。

后端实现要求：

- 在不破坏现有 OpenAPI 输出的情况下补 Knife4j 文档入口。
- `/doc.html` 可访问并展示当前后端 API。
- 保留 `/swagger-ui.html` 或现有 OpenAPI JSON 入口，避免影响现有验证脚本。
- 如依赖冲突导致 Knife4j 不适配当前 Spring Boot 版本，需给出明确说明并提供替代方案，例如 `/doc.html` 重定向到 Swagger UI。

验收标准：

- 后端启动后可访问 `/doc.html`。
- OpenAPI JSON 仍可访问。
- 后端测试通过。

### B2：补 `/api/triage/consult` 兼容接口

对应课程要求：

- 任务三示例路径为 `/api/triage/consult`。

当前问题：

- 当前真实接口为 `/api/triage/analyze`。
- 功能满足，但路径与课程截图不一致。

后端实现要求：

- 新增 `/api/triage/consult` 兼容接口。
- 该接口复用现有分诊服务逻辑，不复制业务实现。
- 请求体和响应结构尽量兼容现有 `/api/triage/analyze`。
- 保留 `conversationId/history/round` 多轮问诊参数流转。
- 不要求本轮持久化完整 AI 会话消息；若要实现，需先单独提出数据库变更方案。

验收标准：

- `/api/triage/consult` 与 `/api/triage/analyze` 在同等输入下返回同等业务结果。
- 原有 `/api/triage/analyze` 不受影响。
- 补充 controller/service 测试或集成测试。

### B3：为分诊推荐医生展示提供稳定数据支持

对应课程要求：

- 任务三要求前端展示推荐科室与医生列表，并能跳转到该医生挂号页。

当前问题：

- 前端可通过现有医生/排班接口间接查询，但分诊结果本身不直接给出推荐医生列表。

后端实现要求：

- 优先复用现有医生、排班、科室接口。
- 如现有接口不足以支持前端“一次性加载推荐医生卡片”，可新增轻量查询接口，例如按科室返回可预约医生及最近排班摘要。
- 不改变分诊 AI 的核心响应语义；推荐医生可以由业务排班数据筛选生成，不必由 AI 直接指定。
- 返回数据应包含医生 ID、科室 ID、医生姓名、职称、可预约日期/排班 ID、剩余号源摘要。

验收标准：

- 前端可根据分诊推荐科室稳定获取 2-3 个可预约医生。
- 无可用医生时返回空列表，不报 500。
- 数据仅来自真实医生和排班表，不使用前端 mock。

### B4：补 `/api/prescription/check` 兼容接口

对应课程要求：

- 任务四示例路径包括 `/api/prescription/check`。
- 医生点击“AI 辅助审核”后展示用药建议、相互作用检测结果和风险提示。

当前问题：

- 当前项目在创建处方时自动触发 AI 审核。
- 医生端按钮主要读取已有处方详情中的审核结果。

后端实现要求：

- 新增 `/api/prescription/check` 兼容接口。
- 支持两种输入方式中的至少一种：
  - 传入 `prescriptionId`，返回该处方已有审核结果；如不存在则触发审核。
  - 传入处方药品列表与患者信息，返回预审核结果但不保存正式处方。
- 推荐优先实现 `prescriptionId` 方式，成本低且不破坏现有流程。
- 返回内容应包含风险等级、建议、警告、规则检查摘要、AI 审核状态。
- 保持确定性规则优先，AI 不得降低确定性规则判定出的风险等级。

验收标准：

- 前端点击 AI 审核可显式调用该接口。
- 高风险处方返回明确风险提示。
- AI 失败时仍返回规则审核结果和降级说明。

### B5：支持医生端问诊对话记录参与 AI 病历生成

对应课程要求：

- 任务五要求医生端可录入问诊对话记录，并由 AI 自动生成结构化病历。

当前问题：

- 当前病历生成主要接收结构化字段。
- 缺少明确的“问诊对话记录”输入字段。

后端实现要求：

- 在病历生成请求 DTO 中增加可选字段，例如 `consultationTranscript` 或 `dialogueText`。
- AI 病历生成 Prompt 中加入该字段，让模型从对话中提取主诉、现病史、既往史、体格检查、初步诊断、治疗建议。
- 字段为空时保持现有结构化字段生成逻辑。
- 生成结果仍保存为 `AI_GENERATED` 草稿，必须医生确认后才变为正式病历。

验收标准：

- 请求中带问诊对话文本时，AI 生成结果可回填结构化病历字段。
- 旧请求不带该字段时仍兼容。
- 有测试覆盖空文本、有效文本、AI 失败降级。

### B6：按科室补 Prompt 工程优化

对应拓展要求：

- 在后端 Service 中设计可配置 Prompt 模板，针对不同科室如儿科、心内科优化病历生成质量。

当前问题：

- 已有 `PromptManager`、`prompts/*_v*.txt` 和 Prompt 版本审计。
- 未见按具体科室拆分的专用 Prompt 模板。

后端实现要求：

- 保留现有 capability 级 Prompt，例如 `medical_record_v1.txt`。
- 增加科室维度的可选 Prompt 选择策略，例如：
  - `medical_record_internal_v1.txt`
  - `medical_record_cardiology_v1.txt`
  - `medical_record_pediatrics_v1.txt`
- 当请求中可识别科室时，优先使用科室专用 Prompt；否则回退到通用 Prompt。
- AI 调用审计中记录实际使用的 Prompt 版本或 Prompt key。
- Prompt 模板应强调 AI 辅助、医生确认、不要编造检查结果。

验收标准：

- 内科/心内科/儿科至少有两个科室专用 Prompt 示例。
- 回退逻辑可用，不因找不到科室模板导致 AI 调用失败。
- 单元测试覆盖 Prompt 选择和回退。

## 四、交付物

- 后端代码改动。
- 兼容接口说明。
- Prompt 模板文件。
- 后端测试结果。
- 如新增接口，补充 OpenAPI/Knife4j 可见性验证。

## 五、联调交接点

交给联调 AI 验证的场景：

- `/doc.html` 可访问。
- `/api/triage/consult` 可返回分诊结果。
- 分诊推荐科室可获取可预约医生列表。
- `/api/prescription/check` 可返回 AI 审核结果。
- 带问诊对话记录的病历生成可完成。
- 按科室 Prompt 调用后审计记录可追踪。
