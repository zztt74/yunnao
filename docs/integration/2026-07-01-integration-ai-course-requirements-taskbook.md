# 联调 AI 任务书：课程要求补强验收

日期：2026-07-01  
分支：`codex/integration`  
依据文档：`docs/integration/2026-07-01-course-requirements-checklist.md`（开头标记：修改后）

## 一、任务目标

联调 AI 负责把前端 AI 和后端 AI 的补强结果串成可验证的课程演示闭环。重点不是新增业务功能，而是确认课程对照文档中的待补强项都被实现、可运行、可演示、可回归。

## 二、工作边界

允许做：

- 更新或新增非侵入式联调脚本。
- 补充课程演示说明、截图清单、验证报告。
- 调整 Docker Compose 或本地启动说明中的联调配置。
- 对前后端提交做契约一致性检查。

不允许做：

- 不替前端 AI 大改页面业务。
- 不替后端 AI 重写核心接口。
- 不擅自修改数据库基线。
- 不跳过真实后端改用 mock 通过演示。

## 三、验收任务清单

### I1：确认任务一 Knife4j 文档入口

覆盖要求：

- `/doc.html` 可访问。
- 原 Swagger/OpenAPI 入口不被破坏。

验证步骤：

- 启动后端或 Docker Compose。
- 访问 `/doc.html`。
- 访问现有 `/swagger-ui.html` 或 OpenAPI JSON。
- 检查主要模块接口是否出现在文档中：分诊、挂号、处方、病历。

通过标准：

- `/doc.html` 返回 200 或可正常重定向到可用 API 文档页。
- OpenAPI 文档仍可用于前后端契约校验。

### I2：确认任务三分诊到医生级挂号闭环

覆盖要求：

- `/api/triage/consult` 兼容接口可用。
- 分诊结果页展示推荐科室和医生列表。
- 用户可一键跳转到该医生挂号页。

验证步骤：

- 患者登录。
- 输入症状发起 AI 分诊。
- 检查分诊结果中是否出现推荐医生卡片。
- 点击“预约该医生”。
- 检查挂号页是否预选或定位到目标医生/科室。
- 完成挂号。

通过标准：

- 分诊结果不是只显示科室，还能看到医生列表。
- 医生级跳转参数生效。
- 无可预约医生时有降级提示。
- 原有按科室挂号路径仍可用。

### I3：确认多轮问诊边界说明

覆盖要求：

- 当前文档说明多轮问诊 `conversationId/history/round` 未持久化完整 AI 会话消息。
- 课程任务三不强制持久化完整会话，但产品化可后续建会话表和消息表。

验证步骤：

- 跑一次二轮问诊脚本或手工操作。
- 确认第二轮请求携带前一轮上下文。
- 确认前端交互不会误导用户“完整长期聊天记录已保存”。

通过标准：

- 二轮问诊可用。
- 报告中明确标注：完整 AI 会话持久化不是本轮课程必需项。

### I4：确认任务四 AI 处方审核闭环

覆盖要求：

- `/api/prescription/check` 兼容接口可用。
- 医生端能看到历史处方列表及对应 AI 审核结果。

验证步骤：

- 医生登录。
- 进入某患者接诊/处方页。
- 创建或查看处方。
- 点击 AI 审核按钮。
- 检查是否调用兼容接口或明确刷新审核结果。
- 查看历史处方面板。

通过标准：

- 审核结果包含风险等级、用药建议、警告或规则检查摘要。
- 历史处方列表能显示 AI 审核风险信息。
- AI 不可用时页面仍有规则审核或降级提示。

### I5：确认任务五 AI 病历生成闭环

覆盖要求：

- 医生端有问诊对话记录多行输入区。
- 问诊对话文本能参与 AI 病历生成。

验证步骤：

- 医生登录。
- 进入接诊/病历页面。
- 输入模拟医患对话文本。
- 点击 AI 生成病历。
- 检查主诉、现病史、既往史、初步诊断、治疗建议等字段是否回填。
- 医生编辑后保存/确认病历。
- 患者端查看已确认病历。

通过标准：

- 对话文本不为空时被传给后端。
- 生成结果可编辑、可保存、可确认。
- 文本为空时原有结构化字段生成流程仍可用。

### I6：确认拓展项 Pinia 状态优化

覆盖要求：

- 前端使用 Pinia 模块化管理患者信息、医生信息、挂号记录、处方记录等状态。

验证步骤：

- 检查 store 文件和页面引用。
- 操作分诊到挂号、医生处方历史、病历生成流程。
- 切换患者或退出登录后检查状态是否清理。

通过标准：

- 状态模块拆分清晰，不只有 `auth` 和 `encounter`。
- 跨页面流转稳定。
- 不出现不同患者数据串扰。

### I7：确认拓展项 Prompt 工程优化

覆盖要求：

- 后端 Prompt 可配置。
- 至少存在按科室优化的 Prompt 模板或选择策略。

验证步骤：

- 检查 `backend/src/main/resources/prompts`。
- 检查 PromptManager 或相关 Service 的科室选择逻辑。
- 触发不同科室的病历生成或调用单元测试。
- 检查 AI 调用审计中 Prompt 版本/Key。

通过标准：

- 至少两个科室专用 Prompt 示例存在。
- 找不到科室专用 Prompt 时可回退通用 Prompt。
- 审计记录能追踪 Prompt 使用情况。

## 四、推荐验证命令

按实际环境调整：

```powershell
docker compose up -d --build
npm --prefix frontend run build
mvn -f backend/pom.xml test
node tests/integration/e2e-real-clinic-flow.mjs
```

如项目已有更准确的启动脚本，优先使用项目脚本。

## 五、最终交付物

联调 AI 需要输出：

- 一份补强验收报告。
- 一份浏览器演示步骤清单。
- 一份失败项/阻塞项清单。
- 每个失败项标明责任方：前端 AI、后端 AI、联调 AI。

## 六、完成判定

全部满足以下条件才算收尾：

- 课程对照文档中所有待补强项都有实现或明确说明。
- 前端构建通过。
- 后端测试通过。
- Docker 或本地真实联调通过。
- 患者端、医生端关键页面可浏览器演示。
- 不存在回退到 mock 才能跑通的关键流程。
