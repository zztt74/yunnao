# 后端 AI 剩余交付任务书

日期：2026-07-01  
分支：`codex/integration`  
目标：补齐当前项目后端缺口，使前端三端主要页面不再出现“后端接口未提供”的阻塞，并支撑完整真实业务闭环验收。

协作角色限定：本项目剩余工作只划分为后端 AI、前端 AI、联调 AI。后端 AI 只负责本文档内的后端实现与后端测试；涉及契约、AI 能力验收、E2E、Docker、最终验收的问题交给联调 AI 协调。

## 一、工作边界

### 允许做

- 基于现有 Spring Boot 模块补控制器、Service、Repository、DTO。
- 基于现有表结构实现缺失接口。
- 补必要的权限校验、状态校验、错误码。
- 补后端单元测试或集成测试。
- 补联调用种子数据脚本所需的后端支持。

### 暂不允许擅自做

- 不直接改 `main`。
- 不擅自修改治理文档、角色任务书。
- 不擅自修改数据库基线或 Flyway 基线脚本；如必须改表结构，先写明原因并交给联调 AI 确认。
- 不擅自扩展医疗业务规则到生产级复杂度，例如收费、库存、医保、真实临床规则。
- 不把任何本地密钥、AI Key、数据库密码写入 Git。

## 三、必须完成的后端任务

### 任务 B1：医生本人资料更新接口

当前问题：前端 `DoctorProfileView` 保存资料时只能报错，因为缺医生本人资料更新接口。

建议接口：

- `PUT /api/doctors/me`

建议逻辑：

- 从当前登录用户解析 `userId`。
- 根据 `doctor.user_id` 找医生档案。
- 允许更新：`phone`、`email` 如当前模型支持；若 Doctor 表无字段，则只更新已有字段：`specialty`、`introduction`、`education`、`experienceYears`。
- 不允许医生自行修改 `departmentId`、`status`、`title`，除非已有业务规则允许。

验收：

- 医生账号登录后能保存个人简介/专长等资料。
- 非医生账号访问返回权限错误。
- 无本地 mock 或假成功。

### 任务 B2：设备档案创建和更新

当前问题：设备管理页面可以查设备、改状态、看历史，但不能新增/编辑设备档案。

建议接口：

- `POST /api/devices`
- `PUT /api/devices/{id}`

建议字段：

- `code`
- `name`
- `category`
- `location`
- `departmentId`
- `applicableItems`
- `status`
- `enabled`

实现要求：

- `code` 唯一。
- 更新时不允许破坏正在使用中的设备状态。
- 设备状态变更仍走现有状态变更接口，基础档案更新不要绕过状态历史。

验收：

- 管理员能新增设备。
- 管理员能编辑设备名称、位置、适用项目等基础信息。
- 新建设备能在设备列表查询到。

### 任务 B3：管理员用户管理基础接口

当前问题：管理端用户管理页面没有真实后端接口。

建议接口：

- `GET /api/admin/users`
- `POST /api/admin/users`
- `PUT /api/admin/users/{id}`
- `POST /api/admin/users/{id}/status`
- `POST /api/admin/users/{id}/reset-password`

第一阶段建议只交付基础用户能力：

- 用户列表分页、按角色/状态/关键字筛选。
- 创建普通账号并绑定角色。
- 更新姓名、手机号、邮箱、角色。
- 启用、禁用、锁定。
- 重置密码。

需要谨慎处理：

- 创建医生账号是否同步创建医生档案。
- 创建患者账号是否同步创建患者档案。

建议第一阶段规则：

- 管理员创建 `ADMIN` 用户：只创建用户账号。
- 管理员创建 `DOCTOR` 用户：如请求带医生档案字段，则同步创建医生档案；否则拒绝并提示必须补齐医生档案字段。
- 管理员创建 `PATIENT` 用户：不建议先做后台创建患者，可先保留患者自注册。

验收：

- 管理端用户列表有真实数据。
- 禁用用户后不能登录。
- 重置密码后可用新密码登录。
- 关键权限必须限制为管理员。

### 任务 B4：管理员全量分诊记录查询

当前问题：分诊只有按患者查询，管理端需要全量列表。

建议接口：

- `GET /api/triage`

建议查询参数：

- `page`
- `pageSize` 或 `size`，需与契约组统一。
- `patientId`
- `priority`
- `departmentId`
- `startDate`
- `endDate`

验收：

- 管理员可分页查看所有分诊记录。
- 可按患者、优先级、日期筛选。
- 非管理员不能全量查看。

### 任务 B5：AI 调用日志分页列表

当前问题：已有 AIInvocation 详情和 attempts，但缺列表接口。

建议接口：

- `GET /api/audit/ai/invocations`

建议查询参数：

- `page`
- `pageSize` 或 `size`
- `capability`
- `success`
- `businessType`
- `startDate`
- `endDate`

验收：

- 管理端 AI 日志页能分页显示真实调用记录。
- 可点开某条记录查看 attempts。
- 不泄露 AI Key、完整敏感请求头。

### 任务 B6：患者总数统计

当前问题：管理端统计总患者数无法真实填充。

建议方案：

- 在 `GET /api/statistics/dashboard` 增加 `totalPatientCount`。
- 或新增 `GET /api/statistics/patients/count`。

验收：

- 管理端 dashboard 不再显示固定 0。
- 统计值来自真实患者表。

## 四、建议补充任务

当前只有 `/api/patients/search`，不支持管理端全量分页。

建议接口：

- `GET /api/patients`

验收：

- 管理员可分页查看患者列表。

### 任务 B8：统一分页参数

当前后端有的接口使用 `pageSize`，有的使用 `size`。

要求：

- 先不要大规模破坏兼容。
- 新接口建议同时兼容 `pageSize` 和 `size`，内部统一转换。
- 最终由契约组决定主参数名。

## 五、与前端 AI、联调 AI 的交接

后端 AI 每完成一个接口，需要给前端 AI 和联调 AI 提供：

1. 接口路径、方法、权限。
2. 请求 DTO 和响应 DTO。
3. 状态机或业务校验规则。
4. 错误码和错误消息。
5. 是否需要种子数据或迁移说明。

联调 AI 负责把这些接口同步到最终联调清单和验收报告。
