## 2. 输入输出 Schema

### 输入（[TriageAIRequest](file:///d:/shixun/yunnao/backend/src/main/java/com/neusoft/cloudbrain/ai/dto/TriageAIRequest.java)）

| 字段 | 类型 | 说明 |
|---|---|---|
| ageRange | String | 年龄区间 |
| gender | String | 性别 |
| chiefComplaint | String | 主诉 |
| duration | String | 症状持续时间 |
| supplement | String | 补充描述 |

不包含患者 ID、姓名、手机号等隐私信息（最小化原则）。

### 输出（[TriageAIResult](file:///d:/shixun/yunnao/backend/src/main/java/com/neusoft/cloudbrain/ai/dto/TriageAIResult.java)）

| 字段 | 类型 | 受控规则 |
|---|---|---|
| departmentCode | String | 不在 AI 侧做枚举校验（业务侧 findByCode 拒绝未知编码，对照 [12_业务流程与状态机.md 第5节](file:///d:/shixun/yunnao/智慧云脑诊疗平台_AI协作文档_v2/product/12_业务流程与状态机.md#L137)） |
| priority | String | AI 侧枚举受控：EMERGENCY / HIGH / MEDIUM / LOW |
| symptomKeywords | List\<String\> | 识别到的关键症状词 |
| reason | String | 推荐理由 |
| safetyNotice | String | 安全提示（含"仅供辅助参考"） |
| emergencySuggested | boolean | 是否建议走急诊通道 |

## 4. Provider 变化

### AITriageServiceImpl 改造

[AITriageServiceImpl](file:///d:/shixun/yunnao/backend/src/main/java/com/neusoft/cloudbrain/ai/application/AITriageServiceImpl.java) 变更：

1. **不在 AI 侧做 departmentCode 枚举校验**（修正）：依据任务卡异常流程"不存在科室 → departmentCode 校验失败 → **业务侧拒绝**"和 [12_业务流程与状态机.md 第5节](file:///d:/shixun/yunnao/智慧云脑诊疗平台_AI协作文档_v2/product/12_业务流程与状态机.md#L137)"科室编码必须映射到真实 Department；映射失败时转人工选择"，由 [TriageService.mapDepartment()](file:///d:/shixun/yunnao/backend/src/main/java/com/neusoft/cloudbrain/triage/service/TriageService.java#L193) 通过 `findByCode` 拒绝未知编码并设置 `mappingStatus=MANUAL`
2. **错误码修正**：
   - 超时/Provider 异常 → `AI_TRIAGE_FAILED`（HTTP 504）
   - 非法 JSON / Schema 校验失败 → `AI_INVALID_RESPONSE`（HTTP 500）
   - 对照任务卡异常流程规范

### MockAIProvider 调整

[MockAIProvider](file:///d:/shixun/yunnao/backend/src/main/java/com/neusoft/cloudbrain/ai/provider/MockAIProvider.java) 变更：

- 空结果场景：从空字符串 `departmentCode:""` 改为受控默认值 `DEPT_INTERNAL` + `LOW`，使空结果仍通过 Schema 校验

## 5. Mock 场景

7 场景全覆盖（对照 [32_AI能力契约规范.md 第4节](file:///d:/shixun/yunnao/智慧云脑诊疗平台_AI协作文档_v2/contracts/32_AI能力契约规范.md)）：

| 场景 | 触发关键词 | 行为 |
|---|---|---|
| 正常结果 | 默认 | 基于医疗关键词路由到对应科室 |
| 高风险结果 | MOCK_HIGH_RISK 或胸痛关键词 | 胸痛→DEPT_CARDIOLOGY + HIGH + emergencySuggested=true |
| 空结果 | MOCK_EMPTY | DEPT_INTERNAL + LOW + 空关键词 + "信息不足" |
| 超时 | MOCK_TIMEOUT | 抛 AIProviderException(retryable=true) |
| 非法 JSON | MOCK_INVALID_JSON | 返回非 JSON 文本 |
| 不存在科室 | MOCK_NOT_EXIST_DEPT | 返回 DEPT_NOT_EXIST_999（校验失败） |
| Provider 异常 | MOCK_PROVIDER_ERROR | 抛 AIProviderException(retryable=false, 500) |

## 6. 超时和降级策略

```text
超时 → 重试 1 次 → 仍失败 → AIInvocation(FAILED) → 抛 AI_TRIAGE_FAILED (HTTP 504) → 业务进手动流程
非法 JSON → AI_INVALID_RESPONSE → 不无限修复 → 降级
不存在科室 → departmentCode 校验失败 → AI_INVALID_RESPONSE → 业务侧拒绝
```

- 重试仅对超时/5xx，非法 JSON 不重试
- 不无限修复模型输出
- 1 次业务调用 = 1 条 AIInvocation + N 条 AIInvocationAttempt

## 8. 对后端业务模块的调用说明

- AI 分诊不直接对外暴露 HTTP，由 triage 业务模块编排调用
- 调用入口：`AITriageService.analyze(TriageAIRequest)`
- 输入不含患者隐私 ID（最小化原则）
- 异常时业务模块捕获 `BusinessException`，按错误码降级：
  - `AI_TRIAGE_FAILED` → 转手动选择科室
  - `AI_INVALID_RESPONSE` → 转手动选择科室

---

## 9. 潜在安全风险

- Prompt 中明确禁止输出医生 ID 和确定性诊断，但真实模型可能不遵守 → 由 JsonSchemaParser 做枚举受控兜底
- 输入已做脱敏（不含患者隐私 ID），但需业务模块确保传入前完成脱敏
- 日志和 AIInvocation 中不保存 API Key（由 AIInvocationRecorder 保证）

---

## 10. 未解决问题

- 受控科室编码目前硬编码在 AITriageServiceImpl 中，若后续科室新增需同步修改 → 后续可考虑从数据库或配置读取
- JSON Schema 校验仍为手写降级方案，是否引入 `com.networknt:json-schema-validator` 待后端角色确认
