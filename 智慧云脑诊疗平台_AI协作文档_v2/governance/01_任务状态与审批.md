# 任务状态与审批

## 1. 状态

```text
DRAFT → READY → IN_PROGRESS → REVIEW → DONE
                    ↕          │
                 BLOCKED ←─────┘
```

| 状态 | 含义 |
|---|---|
| `DRAFT` | 信息不完整，不允许开发 |
| `READY` | 范围、依赖、契约、目录和验收明确，可以领取 |
| `IN_PROGRESS` | 负责人正在执行 |
| `BLOCKED` | 存在当前角色无法独立解除的阻塞 |
| `REVIEW` | 实现和自测完成，等待评审 |
| `DONE` | 评审通过并关闭 |

## 2. 转换规则

| 转换 | 执行人 | 必要条件 |
|---|---|---|
| `DRAFT → READY` | 评审人 | 模板必填项完整；依赖、契约、目录和测试明确 |
| `READY → IN_PROGRESS` | 负责人 | 依赖满足；分支明确；无未批准变更 |
| `IN_PROGRESS → BLOCKED` | 负责人 | 记录原因、责任角色、已完成部分和解除条件 |
| `BLOCKED → IN_PROGRESS` | 负责人 | 阻塞解除并记录证据 |
| `IN_PROGRESS → REVIEW` | 负责人 | 交付完成；适用测试通过；交付说明完整 |
| `REVIEW → IN_PROGRESS` | 评审人 | 记录问题、责任人和复验要求 |
| `REVIEW → DONE` | 评审人 | 验收和证据完整；适用检查通过；无未报告越权 |

## 3. 补充规则

- 多个评审人全部确认后才能进入 `DONE`；
- 负责人不得自行把自己的任务从 `REVIEW` 改为 `DONE`；
- 父任务在全部适用子任务 `DONE` 且联调完成后，由项目负责人批准；
- `DONE` 后继续修改，必须由原评审人重开为 `IN_PROGRESS`；
- 每次转换都追加状态变更记录，不覆盖历史；
- 父任务状态由联调 AI 维护，架构 AI 只审核拆分和边界。
