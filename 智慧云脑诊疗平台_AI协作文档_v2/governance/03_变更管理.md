# 变更管理

## 1. 何时必须提交变更

- 改变项目范围或架构；
- 改变目录所有权或公共文件负责人；
- 改变 URL、HTTP 方法、字段、枚举、错误码或状态转换；
- 改变核心实体关系、数据库通用规则或安全基线；
- 改变 AI Schema 或确定性规则输入；
- 当前任务需要突破角色任务书授权。

纯文字修正且不改变语义时，由文档负责人维护并在 PR 说明。

## 2. 变更流程

```text
使用 changes/CHANGE_TEMPLATE.md 建立 DRAFT
→ 写明当前定义、建议变更和影响范围
→ 受影响角色评审
→ 项目负责人或规定审批人批准
→ 状态改为 APPROVED
→ 归档到对应 approved 目录
→ 联调 AI 更新 OpenAPI 等机器契约
→ 各角色更新实现、Mock 和测试
→ 回归通过后标记 IMPLEMENTED
```

## 3. 审批责任

- 功能范围、架构、长期协作文档和目录所有权：项目负责人、架构 AI 和全部受影响角色；
- URL、字段、枚举、错误码、状态转换：联调、前端、后端；涉及 AI Schema 时增加 AI AI；
- 数据库迁移版本：后端 AI 分配；
- 具体业务契约变更归档：根 `docs/changes/approved/`；
- 协作文档基线变更归档：文档包 `changes/approved/`。

未批准变更不得进入 `approved/`，不得作为实现依据。

## 4. 禁止做法

```text
后端先改字段再通知前端
前端同时兼容新旧字段
直接修改已执行 Flyway
删除历史批准记录
在聊天中口头批准但不记录
```
