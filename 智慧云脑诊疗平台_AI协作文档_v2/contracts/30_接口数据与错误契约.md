# 接口、数据与错误契约

## 1. 契约唯一来源

接口定义以仓库根目录的：

```text
contracts/openapi.yaml
```

为唯一来源。

后端 DTO、前端 TypeScript 类型、Mock Server 和集成测试均以该文件为准。

配套契约文件均位于仓库根目录：

```text
contracts/openapi.yaml
contracts/schemas/
contracts/enums.md
docs/changes/approved/
```

- `contracts/openapi.yaml` 定义 HTTP 接口；
- `contracts/schemas/` 保存可复用 Schema；
- `contracts/enums.md` 汇总跨模块枚举、含义和负责人；
- `docs/changes/approved/` 保存已批准业务契约变更记录。

任何 AI 不得擅自修改：

- URL；
- HTTP 方法；
- 字段名；
- 枚举值；
- 响应结构；
- 错误码。

需要变更时，先提交变更说明，由联调测试与集成角色统一修改契约。

## 3. 统一响应结构

```json
{
  "code": "SUCCESS",
  "message": "操作成功",
  "data": {},
  "traceId": "request-trace-id"
}
```

禁止同时存在多套结构，例如：

```text
code/message/data
success/msg/result
status/error/body
```

只能保留统一结构。

`traceId` 使用请求链路中唯一的非敏感字符串。后端可以接收请求头 `X-Trace-Id`，但只接受 1 至 64 位字母、数字、短横线或下划线；缺失或非法时生成新值，并同时写入响应头 `X-Trace-Id` 和统一响应体。

`traceId` 必须继续传递到 AI 调用记录和关键审计日志，不得包含用户 ID、手机号或其他隐私信息。

## 4. 分页结构

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 100,
  "totalPages": 5
}
```

规则：

- `page` 从 1 开始；
- 所有列表接口统一分页字段；
- 不允许部分接口返回 `rows`，部分返回 `records`；
- 页码、每页数量必须进行边界校验。

## 5. 时间格式

API 使用 ISO 8601：

```text
2026-06-22T14:30:00+08:00
```

禁止混用 Unix 时间戳、自定义中文时间和无时区时间。

基础版本统一时区为 `Asia/Shanghai`：

- API 使用带 `+08:00` 偏移的 ISO 8601；
- Java 时间类型优先使用 `OffsetDateTime`；
- MySQL 连接和应用 JVM 时区必须统一配置为 `Asia/Shanghai`；
- 数据库时间字段精度统一到毫秒；
- 只表示日期的字段使用 `LocalDate` 和 SQL `DATE`；
- 禁止在实体、DTO 和前端分别进行未经约定的时区补偿。

## 7. HTTP 状态码规则

| 场景 | HTTP 状态码 |
|---|---:|
| 成功 | 200 |
| 创建成功 | 201 |
| 参数错误 | 400 |
| 未登录 | 401 |
| 无权限 | 403 |
| 资源不存在 | 404 |
| 业务冲突 | 409 |
| 请求过于频繁 | 429 |
| AI 或上游超时 | 504 |
| 系统异常 | 500 |

业务错误码和 HTTP 状态码应同时使用，不得所有错误都返回 200。

## 8. 数据库契约

数据库变更必须通过 Flyway。版本号按模块预留区间：

```text
V001-V009：公共基础、认证和权限
V010-V019：患者、医生、科室和固定药品字典
V020-V029：排班和挂号
V030-V039：分诊和就诊
V040-V049：检查、检验和病历
V050-V059：处方
V060-V069：设备
V070-V079：统计视图和只读聚合
```

规则：

- 禁止手工修改数据库后不提交迁移脚本；
- 已执行的迁移脚本不得直接修改；
- 新变更必须新增版本；
- 版本号由后端负责人统一分配。
- 同一区间内按实际薄切片递增，不要求一个模块只使用一个迁移文件；
- 迁移文件名必须描述实际变更，不使用 `update.sql`、`fix.sql` 等模糊名称。

## 9. 数据字段约束

- 业务表主键统一使用自增 `BIGINT`，字段名为 `id`；
- 外键统一使用 `<entity>_id`；
- API 使用 camelCase；
- 数据库使用 snake_case；
- 枚举值使用大写英文；
- 金额和剂量不使用浮点型；
- 患者身份证号等敏感字段若非验收必需，不应设计；
- 密码字段只保存哈希值。

### 9.1 通用审计字段

可修改业务表统一包含：

```text
created_at DATETIME(3) NOT NULL
updated_at DATETIME(3) NOT NULL
version BIGINT NOT NULL DEFAULT 0
```

- `created_at` 创建后不得修改；
- `updated_at` 每次业务修改时更新；
- `version` 用于乐观锁和并发冲突识别；
- 创建人、更新人仅在业务需要审计操作者时增加 `created_by`、`updated_by`；
- 纯关联表可以按任务卡说明减少字段，但必须有明确理由。

### 9.2 删除规则

- 不设置所有表通用的 `deleted` 字段；
- 用户、医生、科室、药品和设备使用业务状态启用或停用；
- 挂号、就诊、病历、诊断、处方、检查检验、设备使用、AI 调用和审计日志不得物理删除；
- 只有明确允许删除的临时草稿才能使用 `deleted_at`，并在任务卡中说明；
- 正式业务记录通过取消、作废、停用或新增版本处理。

### 9.3 约束和索引

- 外键名称：`fk_<子表>_<父表>`；
- 唯一约束名称：`uk_<表>_<字段>`；
- 普通索引名称：`idx_<表>_<字段>`；
- 外键字段、业务唯一键和主要筛选字段必须建立索引；
- `user_account.username`、`department.code`、`doctor.user_id`、`patient.user_id`、`drug.code` 必须唯一；
- `Encounter.appointment_id` 必须唯一，保证一个挂号只有一个有效 Encounter；
- 重复挂号必须同时通过数据库唯一约束或等价锁定机制和业务校验防止；
- Device 进行中使用记录必须通过数据库约束、锁或条件更新保证唯一；
- 字符集统一使用 `utf8mb4`，排序规则统一使用 `utf8mb4_0900_ai_ci`；
- 表名使用小写 snake_case，禁止使用 MySQL 保留字。

## 10. 契约变更流程

```text
提出变更原因
→ 说明影响接口和页面
→ 联调角色更新 OpenAPI
→ 后端更新实现
→ 前端重新生成类型
→ 更新 Mock 和测试
→ 通过回归后合并
```

禁止：

```text
后端先改字段
→ 群里通知
→ 前端自行兼容
```

## 11. 认证契约冻结项

登录和权限阶段开始前，任务卡和 OpenAPI 必须明确：

- Access Token 有效期；
- 是否使用 Refresh Token；
- 退出登录的失效方式；
- 连续失败锁定阈值和锁定时长；
- 密码最小规则；
- JWT Claims 名称；
- Token 请求头格式；
- 401 与 403 的错误码；
- 前端清理认证状态的条件。

未冻结上述内容前，不得由前端、后端分别猜测实现。

### 11.1 基础版认证安全基线

基础版本统一采用：

```text
密码哈希：BCrypt，cost=12
Access Token：JWT，有效期 2 小时
Refresh Token：不实现
请求头：Authorization: Bearer <token>
JWT 必需 Claims：sub、roles、tokenVersion、iat、exp、jti
前端存储：Pinia 内存 + sessionStorage
退出登录：清理前端 Token，并由后端递增当前用户 tokenVersion
管理员重置密码或停用账号：递增 tokenVersion，使旧 Token 失效
```

规则：

- JWT 签名密钥必须来自环境变量，长度至少 32 字节；
- 后端每次鉴权校验账号状态和 `tokenVersion`；
- 不在 URL、日志或错误响应中返回 Token；
- `sessionStorage` 中只保存 Access Token 和最小认证信息，不保存密码或患者数据；
- 前端发现 401 时清理认证状态并跳转登录页；403 只展示无权限，不清理有效 Token。

### 11.2 Web 安全基线

- Spring Security 使用无状态 JWT，基础版不使用 Cookie 认证，因此关闭 CSRF 前必须确认所有受保护接口只接受 Bearer Token；
- CORS 只允许环境变量中配置的前端 Origin，不允许 `*` 与凭据组合；
- 登录接口按“用户名 + 来源 IP”限流，默认 5 分钟内最多 10 次失败；
- 连续 5 次密码错误锁定账号 15 分钟；
- 参数使用 Validation 校验，数据库查询使用 JPA 参数绑定；
- 页面输出默认转义，不使用未经清理的 `v-html`；
- 初始管理员通过启动配置创建，仅在管理员不存在时执行；用户名和初始密码来自环境变量，首次登录必须修改密码；
- 用户表需要 `must_change_password` 或等价字段；初始管理员完成密码修改前只能访问修改密码和退出接口；
- 生产或演示环境缺少 JWT 密钥、管理员初始密码或数据库密码时必须拒绝启动；
- 安全参数变更必须走已批准变更流程。

## 12. 状态变更契约

每个改变业务状态的接口必须在 OpenAPI 或任务卡中明确：

```text
来源状态
目标状态
允许角色
数据归属条件
前置业务条件
幂等规则
冲突错误码
审计要求
```

状态转换以 `../product/12_业务流程与状态机.md` 为业务基线，OpenAPI 不得定义额外转换。

## 13. 变更审批规则

- 功能范围、架构和目录所有权变更：项目负责人和全部受影响角色确认；
- URL、字段、枚举、错误码和状态转换变更：联调角色、前端角色和后端角色确认；涉及 AI Schema 时增加 AI 角色确认；
- 纯文字修正且不改变语义：联调角色维护，并在 PR 中说明；
- 未批准变更不得进入 `docs/changes/approved/`，不得作为代码实现依据。
