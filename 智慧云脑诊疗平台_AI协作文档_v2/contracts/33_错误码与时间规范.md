# 错误码与时间规范

## 1. 使用场景

设计接口响应、异常映射、时间字段、日志或前端错误处理时阅读本文件。

完整定义以 `30_接口数据与错误契约.md` 为准，本文件提供实现检查清单。

## 2. 统一响应

```json
{
  "code": "SUCCESS",
  "message": "操作成功",
  "data": {},
  "traceId": "request-trace-id"
}
```

禁止引入第二套 `success/msg/result` 或 `status/error/body`。

## 3. 错误码

- 成功固定使用 `SUCCESS`；
- 失败使用已登记分类加大写下划线后缀；
- HTTP 状态码与业务错误码同时使用；
- 不得所有业务错误都返回 HTTP 200；
- 新分类或新公共错误码需要经过契约变更流程。

常用分类：

```text
AUTH_* PERMISSION_* VALIDATION_*
USER_* PATIENT_* DOCTOR_* DEPARTMENT_*
SCHEDULE_* APPOINTMENT_* TRIAGE_* ENCOUNTER_*
EXAMINATION_* LABORATORY_* MEDICAL_RECORD_*
PRESCRIPTION_* DEVICE_* AI_* FILE_* AUDIT_* SYSTEM_*
```

## 4. traceId

- 只接受 1 至 64 位字母、数字、短横线或下划线；
- 缺失或非法时由后端生成；
- 同时写入响应头 `X-Trace-Id` 和统一响应体；
- 传递到 AI 调用记录和关键审计日志；
- 不包含用户 ID、手机号等隐私信息。

## 5. 时间

- API 使用带 `+08:00` 偏移的 ISO 8601；
- 基础时区为 `Asia/Shanghai`；
- Java 优先使用 `OffsetDateTime`；
- 纯日期使用 `LocalDate` 和 SQL `DATE`；
- 数据库时间精度为毫秒；
- 不混用 Unix 时间戳、无时区时间和自定义中文时间；
- 前后端不得各自进行未约定的时区补偿。

## 6. HTTP 状态码

| 场景 | 状态码 |
|---|---:|
| 成功 | 200 |
| 创建成功 | 201 |
| 参数错误 | 400 |
| 未登录 | 401 |
| 无权限 | 403 |
| 不存在 | 404 |
| 业务冲突 | 409 |
| 请求过多 | 429 |
| 上游超时 | 504 |
| 系统异常 | 500 |
