# 数据库与 Flyway 规范

## 1. 使用场景

后端 AI 在新增表、字段、索引、约束或迁移脚本前必须阅读本文件和相关业务状态机。

核心实体、关系、事务和实现责任先读取 `34_数据库设计基线.md`。详细数据库契约仍以 `30_接口数据与错误契约.md` 的“数据库契约”和“数据字段约束”章节为准；本文件提供实际开发检查顺序。

## 2. 开发顺序

```text
检查当前后端任务卡
→ 检查相关业务状态机
→ 确认 Flyway 版本区间
→ 定义主键、外键、唯一约束和索引
→ 定义审计字段和并发策略
→ 创建新的迁移脚本
→ 更新 Entity 和 Repository
→ 编写约束和集成测试
```

## 3. Flyway 区间

```text
V001-V009：公共基础、认证和权限
V010-V019：患者、医生、科室和固定药品字典
V020-V029：排班和挂号
V030-V039：分诊和就诊
V040-V049：检查、检验和病历
V050-V059：处方
V060-V069：设备
V070-V079：统计视图和只读聚合
```

版本号由后端负责人分配。已执行迁移不得修改，新变化必须新增版本。

## 4. 强制规则

- 主键使用自增 `BIGINT id`；
- 外键使用 `<entity>_id`；
- 数据库字段使用 snake_case；
- 枚举值使用大写英文；
- 可修改业务表包含 `created_at`、`updated_at`、`version`；
- 密码只保存哈希；
- 金额和剂量不使用浮点数；
- 字符集 `utf8mb4`，排序规则 `utf8mb4_0900_ai_ci`；
- 外键、业务唯一键和主要筛选字段建立索引；
- 正式医疗记录不物理删除。

## 5. 需要停止

- 需要改变已批准实体关系；
- Flyway 版本与其他任务冲突；
- 任务卡未说明并发、删除或审计策略；
- 数据库结构与 OpenAPI、状态机不一致；
- 需要修改已执行迁移。

停止后按 `../governance/03_变更管理.md` 处理。
