# 数据库设计基线

## 1. 文档定位

本文是后端 AI 设计业务表和 Flyway 迁移时必须遵守的架构基线。

本文只冻结数据库技术方案、核心实体关系、通用约束、关键事务与并发策略、Flyway 版本区间和后端实现责任。本文不规定每张业务表的全部字段，也不直接提供具体业务迁移脚本；业务字段必须从当前任务、业务状态机和 OpenAPI 推导。

## 2. 已完成的数据库基础

项目已具备：

```text
MySQL 8 单数据库
Spring Boot DataSource 配置
Spring Data JPA
Flyway 迁移入口
V001 用户、角色和用户角色基础表
Docker Compose MySQL 服务
数据库环境变量模板
```

后端 AI 不得另建第二套数据库初始化方式，不得使用 Hibernate `ddl-auto` 代替 Flyway。

## 3. 基础技术规则

| 项目 | 基线 |
|---|---|
| 数据库 | MySQL 8 |
| 存储引擎 | InnoDB |
| 字符集 | `utf8mb4` |
| 排序规则 | `utf8mb4_0900_ai_ci` |
| 时区 | `Asia/Shanghai` |
| 时间精度 | 毫秒 |
| 主键 | 自增 `BIGINT id` |
| API 命名 | camelCase |
| 数据库命名 | snake_case |
| 枚举存储 | 大写英文字符串 |
| 迁移工具 | Flyway |
| ORM 建表策略 | `ddl-auto=validate` |

金额、剂量和比例不得使用浮点类型。密码只保存哈希值，不保存明文。

## 4. 核心实体边界

基础版本至少包含以下实体。后端 AI 可以增加必要明细表，但不得擅自合并或删除核心边界。

### 4.1 认证与人员

```text
UserAccount、Role、UserRole
Patient、Doctor、Department
```

- `Patient` 与账号一对一；
- `Doctor` 与账号一对一；
- 一个医生基础版本主要归属一个科室；
- 用户、医生和科室通过业务状态停用，不物理删除历史主体。

### 4.2 排班、分诊与诊疗

```text
Schedule、Appointment、TriageRecord
Encounter、EncounterDiagnosis
```

- `Appointment` 必须关联患者和排班；
- `Encounter.appointment_id` 唯一，一个挂号最多对应一个就诊；
- 诊断通过 `source=AI_SUGGESTION/DOCTOR` 隔离；
- AI 不得创建或覆盖 `FINAL + DOCTOR` 正式诊断。

### 4.3 检查、检验与病历

```text
ExaminationOrder、ExaminationResult
LaboratoryOrder、LaboratoryResult
MedicalRecord
```

- 申请、结果和病历必须关联 `Encounter`；
- 未审核结果不得作为患者可见的正式结果；
- AI 解读不得修改原始检查检验结果；
- 正式病历不得直接覆盖或物理删除。

### 4.4 处方与确定性用药规则

```text
Prescription、PrescriptionItem、PrescriptionReview
Drug、DrugIngredient、DrugInteractionRule
DrugDosageRule、DrugContraindication
```

- 处方业务状态与 AI 审核状态分开保存；
- 处方明细必须引用系统内固定虚构药品；
- 过敏、相互作用、剂量和禁忌先由后端确定性检查；
- AI 只能解释和补充，不能降低确定性规则命中的风险；
- 已确认处方只能作废，不得物理删除。

### 4.5 设备

```text
Device、DeviceUsage、DeviceStatusHistory
```

- 每次设备状态变化必须新增状态历史；
- 同一设备同一时间只能存在一条进行中的使用记录；
- 使用记录关联 `Encounter`，涉及具体检查时同时关联对应申请；
- 设备使用和状态历史不得物理删除。

### 4.6 AI 与审计

```text
AIInvocation、AIInvocationAttempt、AuditLog
```

- 一次业务 AI 调用只创建一条 `AIInvocation`；
- 每次 Provider 请求或重试创建一条 `AIInvocationAttempt`；
- AI 成功率按 `AIInvocation` 统计，重试不重复进入分母；
- 日志不得保存密码、JWT、API Key 或不必要的患者隐私；
- AI 业务关联使用业务类型和业务记录 ID，不建立指向多个业务表的伪外键。

`MedicalRecordRevision`、`MaintenanceRecord`、药品库存、支付和真实发药不进入基础版本。

## 5. 核心关系

```text
UserAccount 1─1 Patient
UserAccount 1─1 Doctor
Doctor N─1 Department
Doctor 1─N Schedule
Patient 1─N Appointment
Schedule 1─N Appointment
Appointment 1─0..1 Encounter
Encounter 1─N EncounterDiagnosis
Encounter 1─N ExaminationOrder
Encounter 1─N LaboratoryOrder
Encounter 1─0..N MedicalRecord
Encounter 1─N Prescription
Prescription 1─N PrescriptionItem
Drug 1─N PrescriptionItem
Encounter 1─N DeviceUsage
Device 1─N DeviceUsage
Device 1─N DeviceStatusHistory
AIInvocation 1─N AIInvocationAttempt
```

正式病历在基础版本中每个 `Encounter` 只能有一条当前有效的 `CONFIRMED` 记录。是否允许多个草稿由后端任务卡明确，但不得出现多个当前正式版本。

## 6. 通用字段与命名

可修改业务表默认包含：

```text
created_at DATETIME(3) NOT NULL
updated_at DATETIME(3) NOT NULL
version BIGINT NOT NULL DEFAULT 0
```

按业务需要增加操作者、确认、取消时间和原因字段。

- 外键字段：`<entity>_id`；
- 外键名：`fk_<子表>_<父表>`；
- 唯一约束名：`uk_<表>_<字段>`；
- 普通索引名：`idx_<表>_<字段>`；
- 外键、业务唯一键和高频筛选字段必须建立索引；
- `TEXT` 或 `JSON` 字段必须由任务卡说明用途、大小和脱敏方式；
- 不设置所有表通用的 `deleted` 字段。

## 7. 必须落实的数据库约束

| 业务规则 | 数据库要求 |
|---|---|
| 用户名唯一 | `user_account.username` 唯一 |
| 科室编码唯一 | `department.code` 唯一 |
| 医生账号唯一 | `doctor.user_id` 唯一 |
| 患者账号唯一 | `patient.user_id` 唯一 |
| 药品编码唯一 | `drug.code` 唯一 |
| 一个挂号一个就诊 | `encounter.appointment_id` 唯一 |
| 重复挂号 | 业务校验加数据库唯一约束或等价锁定机制 |
| 号源不超卖 | 条件更新已预约数，受影响行数为 0 时返回冲突 |
| 设备不可重复占用 | 条件更新设备状态或等价数据库锁定机制 |
| 当前正式病历唯一 | 唯一约束或事务锁定保证 |
| 正式医疗记录保留 | 通过状态、作废或新版本处理，不物理删除 |

MySQL 不能直接通过普通唯一索引表达的“仅进行中记录唯一”等条件，由后端 AI 在任务卡中选择生成列唯一索引、条件更新或事务锁，并提供并发测试。

## 8. 事务与并发基线

必须在单一事务中完成：

- 创建挂号与扣减号源；
- 取消挂号与恢复号源；
- 取消排班与取消其允许取消的挂号；
- 就诊状态变化时同步更新 Appointment 与 Encounter；
- 占用或释放设备并写入状态历史；
- 确认或作废处方并记录审计；
- 确认正式病历并记录审计。

禁止只使用“先查询再更新”保证号源、设备或正式版本唯一性。必须使用条件更新、乐观锁、悲观锁或数据库唯一约束，并验证并发冲突。

## 9. Flyway 实施顺序

```text
V001-V009  公共基础、认证和权限
V010-V019  患者、医生、科室和固定药品字典
V020-V029  排班和挂号
V030-V039  分诊、就诊和诊断
V040-V049  检查、检验和病历
V050-V059  处方和确定性用药规则
V060-V069  设备、使用记录和状态历史
V070-V079  统计视图和只读聚合
```

AI 调用记录和审计日志由首次依赖它们的业务薄切片创建，具体版本由后端负责人登记。已执行迁移不得修改，新变化必须新增迁移。

## 10. 后端 AI 的实现责任

每个业务薄切片由后端 AI 完成：

```text
根据任务和状态机确定字段
→ 对照 OpenAPI 确定对外数据
→ 分配 Flyway 版本
→ 编写表、约束和索引
→ 实现 Entity 与 Repository
→ 实现事务和并发控制
→ 编写迁移、约束和并发测试
→ 提交数据库变更说明
```

后端 AI 可以细化字段和增加必要明细表，但出现以下情况必须停止：

- 需要改变本文核心实体关系；
- 需要新增基础版本之外的实体；
- 数据库字段与 OpenAPI 或状态机冲突；
- 需要修改已执行迁移；
- 无法在现有状态模型下表达业务规则；
- Flyway 版本与其他任务冲突。

停止后按 `../governance/03_变更管理.md` 提交变更。

## 11. 每次数据库交付要求

```text
新增或修改的表
Flyway 文件及版本号
主外键、唯一约束和索引
事务边界与并发控制
删除和审计策略
初始化数据
迁移与回滚说明
测试命令和结果
对 OpenAPI、前端、AI 和统计的影响
```
