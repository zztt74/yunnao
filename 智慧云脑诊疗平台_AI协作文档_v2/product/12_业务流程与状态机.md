# 业务流程与状态机

## 1. 完整主业务流程

```text
管理员初始化科室、医生、排班和设备
→ 患者注册并完善档案
→ 患者输入症状
→ AI 智能问诊与分诊
→ 后端映射真实科室
→ 患者选择真实医生排班
→ 创建挂号
→ 医生查看待诊队列
→ 医生开始接诊
→ 记录问诊和初步判断
→ 开立检查检验
→ 登记设备使用
→ 录入并审核结果
→ AI 解释检查检验
→ AI 生成病历草稿
→ 医生编辑并确认
→ 医生按需创建处方
→ 存在处方时进行 AI 审核
→ 医生确认或作废处方
→ 完成就诊
→ 患者查看结果
→ 管理员查看统计
```

---

## 2. 账号状态

```text
ENABLED
DISABLED
LOCKED
```

规则：

- DISABLED 和 LOCKED 不能登录；
- 历史数据保留；
- 存在业务记录的账号不能物理删除。
- 管理员可以执行 `ENABLED → DISABLED` 和 `DISABLED → ENABLED`；
- 连续登录失败达到配置阈值时执行 `ENABLED → LOCKED`；
- 锁定阈值和锁定时长必须通过配置统一管理，不得在多处硬编码；
- 锁定到期或管理员重置后执行 `LOCKED → ENABLED`；
- 账号状态变化必须记录操作人、时间和原因。

---

## 3. 排班状态

```text
AVAILABLE
FULL
CANCELLED
COMPLETED
```

状态判定：

- `CANCELLED`：管理员已取消；
- `COMPLETED`：排班结束时间已过去且未取消；
- `FULL`：有效预约数达到最大号源；
- `AVAILABLE`：未取消、未结束且仍有剩余号源。

`AVAILABLE` 和 `FULL` 由时间、取消标记和号源计算，不作为可被任意修改的独立业务事实。

规则：

- 同一医生排班不得重叠；
- 已预约数不能超过容量；
- 已有预约不能直接删除；
- `AVAILABLE` 或 `FULL` 可以由管理员取消为 `CANCELLED`，但必须记录原因并处理已有预约；
- 取消排班与取消其 `BOOKED`、`CHECKED_IN` 挂号必须在同一事务完成；
- 因排班取消的挂号记录 `cancellationSource=SCHEDULE`，且不恢复已取消排班的可预约号源；
- 存在 `IN_PROGRESS`、`WAITING_EXAM` 或 `COMPLETED` 挂号时禁止取消排班；
- `AVAILABLE` 与 `FULL` 随号源占用和恢复自动切换；
- 到达结束时间后转为 `COMPLETED`；
- `CANCELLED` 和 `COMPLETED` 不得恢复为可预约状态；
- 挂号号源更新使用数据库条件更新，条件至少包含“未取消、未过期、已预约数小于容量”；
- 挂号和取消必须在事务内同步号源。

## 4. 挂号状态

```text
BOOKED
CHECKED_IN
IN_PROGRESS
WAITING_EXAM
COMPLETED
CANCELLED
NO_SHOW
```

允许转换：

```text
BOOKED → CHECKED_IN
BOOKED → IN_PROGRESS
BOOKED → CANCELLED
BOOKED → NO_SHOW
CHECKED_IN → IN_PROGRESS
IN_PROGRESS → WAITING_EXAM
WAITING_EXAM → IN_PROGRESS
IN_PROGRESS → COMPLETED
```

禁止：

- COMPLETED 再次接诊；
- IN_PROGRESS 后患者取消；
- 一个挂号存在多个进行中 Encounter；
- 过期或取消排班继续挂号。
- `BOOKED → CHECKED_IN` 为可选签到流程；基础版本允许医生直接执行 `BOOKED → IN_PROGRESS`；
- Appointment 与 Encounter 的 `IN_PROGRESS`、`WAITING_EXAM`、`COMPLETED` 必须在同一业务用例中同步。

---

## 5. 分诊优先级

```text
LOW
MEDIUM
HIGH
EMERGENCY
```

规则：

- EMERGENCY 仅为风险提示；
- AI 分诊不是正式诊断；
- 科室编码必须映射到真实 Department；
- 映射失败时转人工选择。

## 6. 就诊状态

```text
CREATED
IN_PROGRESS
WAITING_EXAM
COMPLETED
CANCELLED
```

规则：

- Encounter 是诊疗中心；
- 诊断、病历、处方、检查和设备使用关联 Encounter；
- 医生只能处理本人 Encounter；
- 已完成就诊不能直接回退。

允许转换：

```text
CREATED → IN_PROGRESS
IN_PROGRESS → WAITING_EXAM
WAITING_EXAM → IN_PROGRESS
IN_PROGRESS → COMPLETED
CREATED → CANCELLED
```

转换约束：

- 一个 Appointment 只能关联一个有效 Encounter；
- 开始接诊时必须同时将 Appointment 更新为 `IN_PROGRESS`；
- 等待检查时必须同时将 Appointment 更新为 `WAITING_EXAM`；
- 完成就诊前必须满足本文规定的病历、医生最终诊断、检查检验和已有处方前置条件；
- 完成就诊必须存在已确认正式病历和至少一条医生确认的最终诊断；
- 处方可不存在；存在处方时，其业务状态必须为 `CONFIRMED` 或 `VOIDED`；
- 任一检查检验处于 `ORDERED`、`IN_PROGRESS` 或 `RESULT_ENTERED` 时不得完成就诊；
- 完成时必须同时将 Appointment 更新为 `COMPLETED`；
- `IN_PROGRESS`、`WAITING_EXAM` 和 `COMPLETED` 不允许取消；
- 状态冲突统一返回 `ENCOUNTER_STATUS_CONFLICT`。

## 7. 诊断类型与来源

诊断类型：

```text
PRELIMINARY
FINAL
```

诊断来源：

```text
DOCTOR
AI_SUGGESTION
```

规则：

- AI 只能产生 `AI_SUGGESTION`，不得产生正式 `FINAL + DOCTOR` 记录；
- 医生确认的最终诊断必须为 `type=FINAL`、`source=DOCTOR`；
- 一个 Encounter 至少有一条医生最终诊断后才能完成；
- AI 原始结果保存在 AI 调用记录，结构化候选诊断保存为 `source=AI_SUGGESTION`；
- 医生诊断保存为 `source=DOCTOR`，两类记录不得互相覆盖；
- 最终诊断变化属于正式医疗事实变更，基础版本完成后不得直接修改。

---

## 8. 病历状态

```text
DRAFT
AI_GENERATED
CONFIRMED
AMENDED
```

规则：

- AI 只能生成 DRAFT 或 AI_GENERATED；
- CONFIRMED 必须由医生完成；
- 基础版本允许 `DRAFT → CONFIRMED`、`AI_GENERATED → CONFIRMED`；
- `DRAFT ↔ AI_GENERATED` 表示医生手工草稿与 AI 草稿来源变化，不代表正式确认；
- `AMENDED` 为扩展版本保留状态，基础版本不得进入；
- 扩展版本启用 `CONFIRMED → AMENDED` 时，必须记录修改人、时间、原因和原版本；
- AI 原始草稿永久保留。

---

## 9. 处方状态与 AI 审核状态

```text
DRAFT
CONFIRMED
VOIDED
```

处方业务状态允许转换：

```text
DRAFT → CONFIRMED
CONFIRMED → VOIDED
```

AI 审核使用独立状态：

```text
NOT_REQUESTED
PENDING
REVIEWED
FAILED
```

AI 审核状态允许转换：

```text
NOT_REQUESTED → PENDING
PENDING → REVIEWED
PENDING → FAILED
FAILED → PENDING
REVIEWED → PENDING
```

规则：

- AI 审核状态不等于处方确认状态；
- 高风险需要二次确认；
- CONFIRMED 后只能作废；
- AI 失败允许医生手工继续，但记录失败。
- 基础版本不实现真实发药，因此不定义 `DISPENSED`；
- 处方确认和 AI 审核记录必须分别保存。

---

## 10. 检查检验状态

```text
ORDERED
IN_PROGRESS
RESULT_ENTERED
REVIEWED
CANCELLED
```

规则：

- 未审核结果不向患者展示；
- AI 解读不能修改原始数据；
- 结果关联具体 Encounter。

允许转换：

```text
ORDERED → IN_PROGRESS
ORDERED → CANCELLED
IN_PROGRESS → RESULT_ENTERED
IN_PROGRESS → CANCELLED
RESULT_ENTERED → REVIEWED
RESULT_ENTERED → IN_PROGRESS
```

转换约束：

- `RESULT_ENTERED → IN_PROGRESS` 仅用于审核退回，必须记录原因；
- `REVIEWED` 后不得直接覆盖原始结果；
- `CANCELLED` 和 `REVIEWED` 为终态；
- 患者只能查看 `REVIEWED` 结果。

---

## 11. 设备状态

```text
AVAILABLE
IN_USE
ABNORMAL
MAINTENANCE
DISABLED
```

典型转换：

```text
AVAILABLE → IN_USE
IN_USE → AVAILABLE
IN_USE → ABNORMAL
AVAILABLE → ABNORMAL
ABNORMAL → MAINTENANCE
MAINTENANCE → AVAILABLE
AVAILABLE → DISABLED
ABNORMAL → AVAILABLE
ABNORMAL → DISABLED
DISABLED → AVAILABLE
```

规则：

- 同一设备不能重复占用；
- 异常、维修、停用设备不可使用；
- 使用结束必须更新状态；
- 历史使用记录保留。
- `MAINTENANCE` 及维修记录属于扩展版本；基础版本只使用 `AVAILABLE`、`IN_USE`、`ABNORMAL`、`DISABLED`；
- `IN_USE → ABNORMAL` 必须先结束当前使用记录并记录异常原因；
- `ABNORMAL → AVAILABLE` 和 `DISABLED → AVAILABLE` 只能由管理员执行并记录原因；
- 设备状态转换冲突返回 `DEVICE_STATUS_CONFLICT`。
- 每次状态转换必须新增 `DeviceStatusHistory`，记录来源状态、目标状态、操作人、时间和原因；
- 状态历史不得物理删除，并用于计算历史有效管理时长。

---

## 12. 权限规则

- 患者只能访问本人数据；
- 医生只能访问本人接诊患者的必要信息；
- 管理员管理基础配置和统计；
- 管理员不得修改医生正式诊断；
- 前端权限仅用于界面；
- 后端必须执行身份、角色和数据归属校验。

---

## 13. 删除和修订规则

不得物理删除：

- 挂号；
- 就诊；
- 正式病历；
- 正式处方；
- 检查检验结果；
- 设备使用记录；
- AI 调用记录；
- 审计日志。

处理方式：

- 停用；
- 取消；
- 作废；
- 逻辑删除；
- 新增修订版本。

---

## 14. AI 降级总规则

```text
AI 超时或错误
→ 记录失败
→ 返回统一 AI 错误
→ 前端结束 loading
→ 显示明确提示
→ 提供手动流程
→ 传统业务继续
```

不允许：

- 无限重试；
- 页面永久等待；
- 失败后自动生成虚假结果；
- 为了演示假装真实 AI 成功。
