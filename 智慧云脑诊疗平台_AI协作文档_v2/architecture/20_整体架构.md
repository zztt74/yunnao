# 整体架构

## 1. 架构结论

系统采用：

```text
前后端分离
+ Spring Boot 3 模块化单体
+ Vue 3 单页应用
+ MySQL 单数据库
+ 统一 AI Gateway
+ Mock/真实 AI 双 Provider
```

不采用微服务。

原因：

- 四人项目周期有限；
- 微服务增加部署、认证、调用链和事务复杂度；
- 多 AI 协作下，多服务会扩大配置和联调风险；
- 当前业务规模不需要分布式架构。

## 3. 前端架构

### 3.1 单项目三端

使用一个 Vue 项目，通过路由区分：

```text
/patient/**
/doctor/**
/admin/**
```

不创建三个独立前端项目。

优点：

- 共用登录、请求和类型；
- 共用 UI 组件；
- 降低部署成本；
- 统一权限和风格；
- 减少 AI 重复生成基础设施。

### 3.2 目录

```text
src/
├── api/
├── components/
├── layouts/
├── modules/
├── router/
├── stores/
├── types/
├── utils/
└── views/
```

业务模块：

```text

├── patient/
├── doctor/
├── schedule/
├── appointment/
├── triage/
├── encounter/
├── examination/
├── medical-record/
├── prescription/
├── device/

```

### 3.3 状态管理

Pinia 仅保存跨页面共享状态：

- 当前用户；
- Token；
- 角色和权限；
- 少量全局字典；
- 当前接诊上下文。

列表数据和表单数据不应全部长期保存到 Pinia。

## 5. 核心业务模块

### 5.1 Auth

负责登录、JWT、角色、接口权限和 Token 失效。

认证解决“用户是谁”，业务模块解决“用户能否访问该数据”。

### 5.2 Patient、Doctor、Department

- Patient 保存患者基本档案；
- Doctor 保存医生档案并关联 User；
- Department 保存真实科室；
- AI 推荐科室必须映射到 Department。

### 5.3 Schedule 与 Appointment

Schedule 表示医生可接诊计划，Appointment 表示患者预约。

创建挂号流程：

```text
检查患者
→ 检查排班
→ 锁定号源
→ 创建挂号
→ 更新已预约数
→ 记录日志
```

并发控制是核心。基础版本统一使用数据库条件更新扣减号源，更新条件至少包含“排班有效且已预约数小于容量”；受影响行数为 0 时返回号源冲突。不得由各任务分别选择不同并发方案。

### 5.4 Triage

```text
患者症状
→ AI 分诊
→ 校验结构
→ 映射真实科室
→ 查询排班
→ 保存分诊记录
```

AI 不生成真实医生 ID。

### 5.5 Encounter

Encounter 是一次实际就诊，是系统核心聚合对象。

关联：

```text
患者
医生
挂号
分诊
正式诊断
病历
检查检验
处方
设备使用
```

不能用 Appointment 代替 Encounter。

`EncounterDiagnosis` 保存医生确认的初步或最终诊断：

```text
encounterId
diagnosisName
diagnosisCode
type
source
confirmedBy
confirmedAt
```

其中 `type` 为 `PRELIMINARY` 或 `FINAL`，`source` 为 `DOCTOR` 或 `AI_SUGGESTION`。AI 原始结果保存在 AI 调用记录，结构化候选可写入 `source=AI_SUGGESTION` 的独立记录，但不得创建或覆盖 `FINAL + DOCTOR` 记录。

### 5.6 Examination 与 Laboratory

基础版本实现：

- 检查申请；
- 检验申请；
- 结果录入；
- 结果审核；
- 患者查询；
- AI 辅助解读。

不接真实设备和真实 PACS/LIS。

### 5.7 MedicalRecord

```text
问诊内容
→ AI 生成草稿
→ 医生编辑
→ 医生确认
→ 正式病历
```

基础版本保存 AI 原始草稿、医生最终版本、生成时间和确认人。病历修订历史属于扩展版本。

### 5.8 Prescription

Prescription 为主表，PrescriptionItem 为药品明细。

基础版本由 Prescription 模块维护固定虚构药品字典和确定性规则，不接入真实药品库、库存或收费。

```text
Drug：编码、名称、规格、启用状态
DrugIngredient：药品成分和过敏匹配标识
DrugInteractionRule：药品组合和风险等级
DrugDosageRule：单次剂量、频次和疗程边界
DrugContraindication：禁忌条件和风险等级
```

后端先执行规则校验，AI 再解释风险和给出辅助建议。确定性规则结果优先于 AI 输出。

PrescriptionReview 保存：

- 风险等级；
- 相互作用；
- 过敏提示；
- 剂量提示；
- 建议；
- AI 状态。

AI 审核状态和处方确认状态分离。

### 5.9 Device

Device 保存设备，DeviceUsage 保存使用记录，DeviceStatusHistory 保存状态变化。基础版本支持异常和停用状态；MaintenanceRecord 和完整维修流程属于扩展版本。

基础版本设备必须实现状态流转、占用校验和使用记录。

### 5.10 Statistics

提供：

- 今日挂号量；
- 今日完成就诊量；
- 科室门诊量；
- 医生接诊量；
- 挂号取消率；
- 患者分布；
- 设备使用率；
- AI 成功率。

统计模块只读。

固定口径：

- 挂号取消率分母包含统计期内创建后又取消的挂号；
- AI Provider 内部重试不重复计算业务调用次数；
- 设备使用率分母由 `DeviceStatusHistory` 计算统计期内 `AVAILABLE` 和 `IN_USE` 的有效管理时长；
- 按日统计统一使用 `Asia/Shanghai` 自然日边界。

## 6. AI 架构

### 6.1 AI Gateway

```text
   ▼

   │
   ▼

├── Prompt Loader
├── Input Sanitizer
├── Provider
├── Response Parser
├── Schema Validator
└── Invocation Logger
```

### 6.2 Provider

```text
MockAIProvider
HttpLLMProvider
```

业务模块不感知厂商。

### 6.3 AI 能力

```text
AITriageService
AIDiagnosisService
AIMedicalRecordService
AIPrescriptionReviewService
AIResultInterpretationService
```

### 6.4 失败处理

```text
请求超时
→ 有限重试
→ 返回统一错误
→ 保存失败记录
→ 业务进入手动模式
```

### 6.5 AI 调用记录

记录业务类型、业务记录 ID、Provider、模型、请求时间、耗时、成功或失败、错误类型和结构化结果。

不得记录 API Key、密码、JWT 和不必要的完整隐私数据。

## 8. 关键流程

### 8.1 分诊与挂号

```text
患者登录
→ 输入症状
→ 后端调用 AI
→ AI 返回科室编码
→ 后端映射科室
→ 查询排班
→ 患者选择排班
→ 事务检查号源
→ 创建挂号
```

### 8.2 医生接诊

```text
医生查看待诊
→ 开始接诊
→ 创建或激活 Encounter
→ 查看患者历史
→ 记录问诊
→ 开立检查检验
→ 生成病历
→ 确认最终诊断
→ 按需创建处方
→ 完成就诊
```

### 8.3 病历生成

```text
提交问诊内容
→ 整理最小输入
→ AI 返回结构化草稿
→ Schema 校验
→ 保存草稿
→ 前端回填
→ 医生编辑确认
```

### 8.4 处方审核

```text
创建处方草稿
→ 调用 AI 审核
→ 返回风险与建议
→ 高风险告警
→ 医生修改或确认
→ 保存正式处方
```

### 8.5 AI 降级

```text
AI 超时或非法响应
→ 后端返回明确错误
→ 前端停止 loading
→ 提示用户
→ 提供手动入口
→ 传统业务继续
```

## 10. 架构治理与四个开发 AI 的所有权

| 角色 | 独占范围 | 不得直接修改 |
|---|---|---|
| 整体架构设计 AI | 协作文档、整体架构、角色边界、目录结构 | 具体业务实现、OpenAPI 实体契约、角色测试 |
| 后端 AI | 业务后端、数据库、后端测试 | 前端、AI Provider、OpenAPI、CI |
| 前端 AI | Vue、状态、路由、前端构建 | 后端、数据库、OpenAPI、Prompt |
| AI 集成 AI | AI 接口、Provider、Prompt、Mock | 传统业务、前端、OpenAPI |
| 联调测试 AI | OpenAPI、契约测试、E2E、CI、部署 | 业务实现、前端页面、Prompt |

跨边界修改必须提交变更申请。

整体架构设计 AI 负责定义边界和审核结构，不负责代替四个开发 AI 实现功能。联调测试 AI 维护父任务验收、OpenAPI 和集成资产，不负责修改长期角色治理规则。

## 11. 实际开发顺序

```text
阶段 0：骨架、契约、Mock
阶段 1：登录和权限
阶段 2：患者、医生、科室
阶段 3：排班
阶段 4：挂号
阶段 5：AI 分诊
阶段 6：医生接诊
阶段 7：检查检验与设备
阶段 8：病历生成
阶段 9：处方审核
阶段 10：统计和可视化
阶段 11：回归和演示环境
```

每个阶段：

```text
冻结当前契约
→ 前后端并行
→ AI 模块提供能力或 Mock
→ 联调当前闭环
→ 自动测试
→ 合并
```

不能等全部前端和后端写完后再统一联调。

### 11.1 阶段 0 必须交付

阶段 0 由一个父任务和四个角色子任务组成：

- 后端：可启动的 Spring Boot 3 模块化单体骨架、Flyway 配置和测试入口；
- 前端：可启动的 Vue 3 单项目骨架、三端路由边界、统一 API Client 和认证存储边界；
- AI：AI Service 接口、`MockAIProvider`、配置边界和 Mock 测试；
- 联调：合法的 OpenAPI 根契约、公共 Schema、API Mock Server、CI、Docker Compose、`.env.example` 和启动说明。

阶段 0 不冻结具体医疗业务字段，不创建伪业务接口。公共响应、分页、时间、错误码、数据库通用规则、安全基线和目录所有权必须先通过父任务确认。

## 12. 关键架构决策

### 模块化单体

降低部署、认证和事务复杂度。

### 单一 OpenAPI

横向分工下，必须有机器可验证的契约。

### Encounter 作为诊疗中心

挂号只是预约，不能承载完整诊疗链路。

### 统一 AI Gateway

隔离厂商，支持 Mock、真实调用和降级。

### AI 只生成建议或草稿

避免模型输出直接成为医疗事实。

### 逐功能联调

把问题提前暴露，避免末期集中爆发。

### 影像 AI 不作为基础实现

缺少数据、训练和临床验证条件，只适合作为外部接口扩展。
