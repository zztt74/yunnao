# 模块与依赖边界

## 1. 架构原则

采用：

```text
Vue 3 单页应用
+ Spring Boot 3 模块化单体
+ MySQL 单数据库
+ 统一 AI Gateway
+ Mock/真实 AI 双 Provider
```

不采用微服务、服务注册中心和分布式事务。

## 2. 前端模块

```text
auth
patient
doctor
admin
schedule
appointment
triage
encounter
examination
medical-record
prescription
device
statistics
```

每个业务模块建议包含：

```text
api/
components/
pages/
stores/
types/
routes.ts
```

原则：

- 页面不能直接拼装 Axios；
- 模块 API 通过统一 Client 调用；
- 共享状态才进入 Pinia；
- 列表和表单临时状态保留在页面或模块内部；
- API 类型由 OpenAPI 生成或校验。

## 4. 后端调用规则

允许：

```text
Controller → Application Service
Application Service → Domain Service
Application Service → Repository
Application Service → 其他模块公开 Service
```

禁止：

```text
Controller → Repository
模块 A → 模块 B 的 Repository
业务模块 → 第三方 AI HTTP Client
统计模块 → 修改业务记录
```

## 5. 核心模块边界

### 5.1 Auth

负责登录、JWT、用户身份、角色权限和 Token 失效处理，不负责具体患者或医生业务权限。

### 5.2 Patient

负责患者基本档案，不承载挂号、病历和处方明细。

### 5.3 Doctor 与 Department

负责医生业务档案和科室信息。

### 5.4 Schedule

负责排班、号源上限、已预约数量和排班冲突。

### 5.5 Appointment

负责患者预约、取消、可选签到和挂号状态。基础版本允许医生从 `BOOKED` 直接开始接诊。

### 5.6 Triage

负责编排 AI 分诊、科室映射、推荐结果保存。

### 5.7 Encounter

负责一次实际就诊，是诊断、病历、检查、处方和设备使用的中心。

`EncounterDiagnosis` 保存结构化候选和医生诊断，通过 `source=AI_SUGGESTION/DOCTOR` 隔离。AI 原始结果保存在 AI 调用记录，AI 不得创建 `FINAL + DOCTOR` 记录。

### 5.8 Examination 与 Laboratory

负责检查申请、检验申请和结果。

### 5.9 MedicalRecord

基础版本负责病历草稿、AI 草稿和医生确认。修订记录属于扩展版本。

### 5.10 Prescription

负责处方主表、药品明细、固定虚构药品字典、确定性用药规则、AI 审核记录和医生确认。处方业务状态与 AI 审核状态分别保存。

基础版用药规则至少包括：

```text
Drug
DrugIngredient
DrugInteractionRule
DrugDosageRule
DrugContraindication
```

处方模块先执行确定性规则检查，再调用 AI 解释风险。AI 输出不得降低或覆盖规则命中的风险。

### 5.11 Device

基础版本负责设备、状态历史、使用记录、异常和停用。故障维修明细属于扩展版本。

### 5.12 Statistics

只读聚合业务数据，不允许修改业务状态。

### 5.13 Audit

负责关键操作日志和追踪。

## 6. AI 模块边界

统一接口：

```text
AITriageService
AIDiagnosisService
AIMedicalRecordService
AIPrescriptionReviewService
AIResultInterpretationService
```

Provider：

```text
MockAIProvider
HttpLLMProvider
```

业务模块只能依赖 AI Service 接口，不得直接调用第三方模型 SDK、自行读取 Prompt、自行解析厂商响应、写死模型名称或保存 API Key。

### 6.1 两类 Mock 的边界

`MockAIProvider`：

- 负责人：AI 能力集成角色；
- 位置：后端 `ai` 模块；
- 用途：在不配置真实 AI Key 时模拟 AI 能力；
- 调用方：后端业务模块；
- 不对前端直接提供另一套接口。

OpenAPI API Mock Server：

- 负责人：联调测试与集成角色；
- 位置：`contracts/`、`tests/contract/` 或联调工具配置；
- 用途：后端接口尚未完成时，按 OpenAPI 向前端提供 HTTP 模拟；
- 调用方：前端；
- 对应真实接口联调通过后必须关闭该接口的 Mock。

禁止前端、后端业务模块或 AI 模块各自创建第三套 Mock API。

## 7. 跨模块示例

### 7.1 创建挂号

```text
AppointmentApplicationService
→ PatientQueryService 检查患者
→ ScheduleCommandService 检查并扣减号源
→ AppointmentRepository 保存挂号
→ AuditService 记录
```

### 7.2 AI 分诊

```text
TriageApplicationService
→ AITriageService 获取建议
→ DepartmentQueryService 映射真实科室
→ ScheduleQueryService 查询可预约医生
→ TriageRecordRepository 保存记录
```

### 7.3 完成就诊

```text
EncounterApplicationService
→ 检查病历确认状态
→ 检查医生最终诊断
→ 检查未完成检查检验
→ 存在处方时检查处方状态
→ 更新 Encounter
→ 更新 Appointment
→ AuditService 记录
```

## 8. 文件存储

报告附件上传属于扩展功能。启用该功能时，使用本地文件目录，并通过 `FileStorageService` 抽象。

数据库只保存文件路径、原始文件名、文件类型、文件大小、上传时间和业务关联 ID，不直接保存大文件二进制。
