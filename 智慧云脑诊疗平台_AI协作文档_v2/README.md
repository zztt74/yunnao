# 智慧云脑诊疗平台 AI 协作文档包 v2.5

## 1. 文档定位

本目录为一个整体架构设计 AI 和四个开发 AI 提供执行依据、权限边界和交付标准。

整理原则：

- 子 AI 从实际开发场景进入，不再每次通读全部共享文档；
- 一份文档承担一种主要职责；
- 角色任务书定义长期职责，任务卡定义本次工作；
- OpenAPI、数据库迁移、Prompt、Mock、测试和公共文件均有唯一负责人；
- 发现冲突时停止并走变更流程，不在实现中兼容两套定义。

## 2. 从哪里开始

任何角色先读：

1. `00_开始这里.md`
2. `01_场景化阅读导航.md`
3. 自己的角色任务书
4. 当前父任务和角色子任务

然后根据实际开发场景读取专项文档。

## 4. 各目录职责

| 目录 | 只回答什么 |
|---|---|
| `governance/` | 文档冲突、任务状态、目录所有权和变更审批 |
| `product/` | 项目做什么、业务要求和状态如何流转 |
| `architecture/` | 系统如何拆分、模块如何依赖、代码放在哪里 |
| 文档包 `contracts/` | HTTP、数据库、错误码、时间和 AI 能力如何约定 |
| `roles/` | 每个 AI 长期负责和禁止什么 |
| `workflow/` | 如何编码、测试、提交、评审和交付 |
| `tasks/` | 当前任务具体做什么 |
| `changes/` | 协作文档基线如何改变 |

## 5. 角色

1. 整体架构设计 AI：治理文档、架构、角色边界和项目结构；
2. 后端开发 AI：Spring Boot 业务、数据库、权限、事务和统计；
3. 前端开发 AI：Vue 页面、路由、状态和交互；
4. AI 能力集成 AI：AI Service、Provider、Prompt、Mock 和降级；
5. 联调测试与集成 AI：OpenAPI、契约测试、E2E、CI 和部署。

整体架构设计 AI 不替代四个开发角色实现具体功能。

## 6. 开发方式

每个业务薄切片建立：

```text
一个父任务
+ 后端子任务
+ 前端子任务
+ AI 子任务（不涉及时写明不适用）
+ 联调子任务
```

执行顺序：

```text
明确业务用例
→ 冻结契约
→ 各角色并行开发
→ 联调当前闭环
→ 正常和异常场景通过
→ 评审与合并
```

父任务记录统一契约版本、依赖和总体验收；角色子任务只修改授权目录。

## 7. 变更记录位置

- 文档包 `changes/approved/`：协作文档、范围和架构基线变更；
- 仓库根 `docs/changes/approved/`：项目实施后的具体接口和业务契约变更。

未批准记录不得作为实现依据。

机器可验证接口契约始终位于仓库根 `contracts/openapi.yaml`；文档包中的 `contracts/` 只保存设计规范和开发指南。
