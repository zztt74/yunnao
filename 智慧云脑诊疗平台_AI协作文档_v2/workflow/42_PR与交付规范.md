# PR 与交付规范

## 1. 提交前

负责人确认：

```text
只修改任务卡允许目录
实现与冻结契约一致
适用测试已运行
交付说明已填写
跨角色影响已报告
没有敏感信息
没有无关重构
```

## 2. PR 必填内容

- 任务编号和目标；
- 修改范围和主要文件；
- 相关接口和 OpenAPI 版本；
- 相关数据表和迁移脚本；
- 测试命令和结果；
- 截图或请求响应证据；
- 是否涉及契约或配置变化；
- 对其他角色的影响；
- 遗留问题；
- 回滚方式。

## 3. 最低检查

根据任务类型执行：

```text
OpenAPI 校验
后端编译、单元和集成测试
前端测试、类型检查和构建
契约测试
基础 E2E
目录所有权检查
敏感信息扫描
Compose 或部署配置检查
```

不适用项必须在任务卡和 PR 中写明原因，并由对应评审人确认。

## 4. 评审与合并

- 负责人提交 `REVIEW`；
- 评审人按任务卡验收，不以“代码能编译”代替需求和契约检查；
- 发现问题退回 `IN_PROGRESS`，记录复验要求；
- 所有评审人确认后进入 `DONE`；
- 父任务只有在角色子任务完成和联调通过后才能关闭；
- PR 应小而明确，不混入无关功能。
